Chapter 1 — Scope & Closures
JS1

What is the difference between var, let, and const? Explain hoisting and the Temporal Dead Zone (TDZ).

Hoisting = JS moves declarations to the top of their scope at compile time — but only the declaration, not the value. TDZ = the period between entering a block and the let/const declaration line where accessing the variable throws a ReferenceError.

Featurevarletconst
ScopeFunctionBlockBlock
Hoisted?Yes (as undefined)Yes (TDZ — unusable)Yes (TDZ — unusable)
Re-declareYesNoNo
Re-assignYesYesNo
Window propertyYesNoNo
JavaScript — hoisting & TDZ
// var is hoisted and initialized as undefined
console.log(x); // undefined — no error
var x = 5;

// let/const are hoisted but in TDZ — unusable
console.log(y); // ReferenceError: Cannot access 'y' before initialization
let y = 10;

// var leaks out of blocks
if (true) {
  var leaked = 'I leak!';
  let safe   = 'block scoped';
}
console.log(leaked); // 'I leak!'
console.log(safe);   // ReferenceError

// Classic loop bug with var
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i)); // 3, 3, 3 — bug!
}
for (let j = 0; j < 3; j++) {
  setTimeout(() => console.log(j)); // 0, 1, 2 — correct
}

// const — binding is immutable, value is not
const user = { name: 'Alice' };
user.name = 'Bob';  // OK — mutation of the object is fine
user = {};          // TypeError — can't reassign the binding

Key interview line

"I always default to const, use let when reassignment is needed, and never use var — it leaks out of blocks and causes subtle loop bugs due to function scoping. The TDZ is actually a feature: it catches use-before-declare bugs that var silently hides as undefined."

JS2

What are Closures? How do they work and what are their practical use cases?

A closure is a function that retains access to variables from its lexical scope even after that outer function has returned. The inner function holds a live reference — not a copy — to the outer scope's variables.

JavaScript — closures in practice
// Basic closure — count persists across calls
function makeCounter() {
  let count = 0;
  return () => ++count;
}
const counter = makeCounter();
counter(); // 1
counter(); // 2
counter(); // 3 — count persists!

// Use case 1: Private state (no class needed)
function createBankAccount(initial) {
  let balance = initial;
  return {
    deposit:    (n) => (balance += n),
    withdraw:   (n) => (balance -= n),
    getBalance: ()  => balance,
  };
}
const acc = createBankAccount(1000);
acc.deposit(500);
acc.getBalance(); // 1500
acc.balance;      // undefined — truly private

// Use case 2: Memoization
function memoize(fn) {
  const cache = {};
  return (...args) => {
    const key = JSON.stringify(args);
    if (cache[key] !== undefined) return cache[key];
    return (cache[key] = fn(...args));
  };
}

// Use case 3: Partial application via currying
const multiply = (a) => (b) => a * b;
const double   = multiply(2); // closes over a=2
double(5);  // 10
double(7);  // 14

Key interview line

"Closures are what make the module pattern, memoization, and partial application possible. The key insight is that a closure captures a live reference to variables — so mutations are reflected. I use closures daily: factory functions, private state without classes, and React hooks (useState/useEffect) are built entirely on closure semantics."

Chapter 2 — Prototypes & OOP
JS3

How does Prototypal Inheritance work? Explain the prototype chain, Object.create(), and how ES6 classes relate.

Every JS object has an internal [[Prototype]] link to another object. Property lookups walk this prototype chain until the property is found or null is reached. ES6 classes are syntactic sugar — typeof MyClass is still "function".

JavaScript — prototype chain & classes
// Prototype chain via Object.create()
const animal = { breathes: true };
const dog    = Object.create(animal);
dog.barks = true;

dog.barks;    // true  — own property
dog.breathes; // true  — from prototype
dog.hasOwnProperty('breathes'); // false

// ES6 Class (syntactic sugar over prototypes)
class Animal {
  constructor(name) { this.name = name; }
  speak() { return `${this.name} makes a sound`; }
}

class Dog extends Animal {
  speak() { return `${this.name} barks! ` + super.speak(); }
}

const d = new Dog('Rex');
d.speak(); // 'Rex barks! Rex makes a sound'

// Proof — classes are prototypes under the hood
typeof Dog;                 // "function"  — not a class type!
typeof Dog.prototype.speak; // "function"

// Chain: d → Dog.prototype → Animal.prototype → Object.prototype → null
d instanceof Dog;    // true
d instanceof Animal; // true
d instanceof Object; // true

Key interview line

"JavaScript uses prototypal inheritance — ES6 classes are purely syntactic sugar. I prefer composition over inheritance in modern code to avoid deep class hierarchies. Object.create() gives you clean prototypal delegation without constructor functions."

JS6

Explain the this keyword. How is it determined, and how do arrow functions differ?

this is determined by how a function is called, not where it is defined. Arrow functions are the exception — they have no own this and inherit it from their lexical scope at definition time.

ContextValue of this
Global scopewindow / global / undefined (strict)
Method call obj.fn()obj
Regular function callundefined (strict) or global
new keywordNewly created instance
Arrow functionLexically inherited — fixed at definition
call / apply / bindExplicitly provided value
JavaScript — this in practice
// Method call — this = obj
const obj = {
  name: 'Alice',
  greet() { console.log(this.name); }
};
obj.greet();   // 'Alice'
const fn = obj.greet;
fn();          // undefined — lost this!

// Arrow function fixes lost-this in callbacks
class Timer {
  constructor() { this.seconds = 0; }
  start() {
    // ❌ regular function — this is undefined in strict mode
    setInterval(function() { this.seconds++; }, 1000);

    // ✅ arrow — inherits this from start()
    setInterval(() => { this.seconds++; }, 1000);
  }
}

// Gotcha: never use arrow functions as object methods
const bad = {
  name: 'Bob',
  greet: () => console.log(this.name) // this = outer scope, not bad!
};

Key interview line

"this is set at call time, not definition time — except for arrow functions, which capture it from their lexical scope permanently. I use arrow functions for callbacks inside methods, but never as object methods themselves since they can't bind to the object."

JS7

What is the difference between call(), apply(), and bind()? When would you use each?

All three explicitly set this. call — invoke immediately, args as comma-separated values. apply — invoke immediately, args as an array. bind — returns a new function with this permanently fixed, does NOT invoke.

JavaScript — call / apply / bind
function greet(greeting, punct) {
  return `${greeting}, ${this.name}${punct}`;
}
const user = { name: 'Alice' };

// call — invoke now, spread args
greet.call(user, 'Hello', '!'); // 'Hello, Alice!'

// apply — invoke now, args as array
greet.apply(user, ['Hi', '?']); // 'Hi, Alice?'

// bind — returns new function, doesn't invoke
const boundGreet = greet.bind(user, 'Hey');
boundGreet('.'); // 'Hey, Alice.'

// Real use case 1: borrow array methods for array-like objects
const args = { 0: 'a', 1: 'b', length: 2 };
Array.prototype.slice.call(args); // ['a', 'b']

// Real use case 2: bind for class event handlers
class Button {
  constructor() {
    this.count = 0;
    this.handleClick = this.handleClick.bind(this);
  }
  handleClick() { this.count++; }
}

// Real use case 3: apply for spread-unfriendly APIs (legacy)
const nums = [3, 1, 4, 1, 5];
Math.max.apply(null, nums); // 5  — modern: Math.max(...nums)

Key interview line

"call and apply both invoke immediately — the only difference is argument format. bind is for saving a function with a fixed context for later use, like passing a class method as a callback. In modern JS, arrow functions and spread reduce the need for call/apply, but bind still shines for class method binding and partial application."

Chapter 3 — Async JavaScript
JS4

Explain the JavaScript Event Loop — Call Stack, Microtasks, and Macrotasks.

JS is single-threaded. The Event Loop allows non-blocking async operations. Microtasks (Promises, queueMicrotask) are fully drained before any macrotask (setTimeout, I/O) runs.

PriorityTypeExamples
1stCall Stack (sync)All synchronous code
2ndMicrotasksPromise.then, queueMicrotask, MutationObserver
3rdMacrotaskssetTimeout, setInterval, I/O callbacks
JavaScript — execution order
console.log('1 — sync');
setTimeout(() => console.log('2 — macrotask'), 0);
Promise.resolve().then(() => console.log('3 — microtask'));
queueMicrotask(() => console.log('4 — microtask'));
console.log('5 — sync');

// Output: 1 → 5 → 3 → 4 → 2
// All microtasks drain before any macrotask runs

setTimeout(() => console.log('timeout'), 0);
Promise.resolve()
  .then(() => { console.log('p1'); return Promise.resolve(); })
  .then(() => console.log('p2'))
  .then(() => console.log('p3'));

// Output: p1 → p2 → p3 → timeout

Key interview line

"The key insight is that microtasks (Promises) always fully drain before any macrotask runs — even a setTimeout(0). Blocking the call stack with heavy sync computation starves both queues and freezes the UI."

JS5

Explain Promises — states, chaining, combinators, and how async/await works under the hood.

A Promise represents an eventual value. States: Pending → Fulfilled | Rejected (transition is irreversible). async/await is syntactic sugar — an async function always returns a Promise; await pauses execution without blocking the thread.

JavaScript — Promise combinators
const p1 = Promise.resolve(1);
const p2 = Promise.resolve(2);
const pFail = Promise.reject('error');

// all — ALL must resolve; fails fast on any rejection
Promise.all([p1, p2]).then(([a, b]) => console.log(a, b)); // 1 2
Promise.all([p1, pFail]).catch(e => console.log(e));      // 'error'

// allSettled — waits for ALL, never rejects itself
Promise.allSettled([p1, pFail]).then(results =>
  results.forEach(r => console.log(r.status, r.value ?? r.reason))
);

// race — first to settle (resolve OR reject) wins
Promise.race([
  new Promise(res => setTimeout(() => res('slow'), 500)),
  new Promise(res => setTimeout(() => res('fast'), 100)),
]).then(v => console.log(v)); // 'fast'

// any — first to RESOLVE wins, ignores rejections
Promise.any([pFail, p1, p2]).then(v => console.log(v)); // 1

// ❌ Sequential — each waits for the previous (total: 2s)
const a = await fetchA();
const b = await fetchB();

// ✅ Parallel — both fire at the same time (total: ~1s)
const [a, b] = await Promise.all([fetchA(), fetchB()]);
CombinatorResolves whenRejects whenUse case
Promise.allAll resolveAny rejects (fast fail)Parallel — all must succeed
Promise.allSettledAll settleNeverNeed all results regardless
Promise.raceFirst settlesFirst settles as rejectionTimeout pattern
Promise.anyFirst resolvesAll rejectFirst success from many sources

Key interview line

"A critical mistake I see is awaiting inside loops sequentially when requests are independent — always use Promise.all for parallel execution. I use allSettled when I need all results regardless of failures, race for timeout patterns, and any for first-success-from-multiple-sources scenarios."

Chapter 4 — Functional Programming
JS8

What is Currying and Function Composition? How do they help write cleaner code?

Currying = transforming a function of N args into a chain of N single-argument functions, enabling partial application. Pipe composes functions left-to-right; compose runs right-to-left.

JavaScript — currying & pipe
// Currying — partial application
const curriedAdd = (a) => (b) => (c) => a + b + c;
curriedAdd(1)(2)(3); // 6

const add10 = curriedAdd(10); // bakes in a=10
add10(5)(3);              // 18

// Generic curry utility
const curry = (fn) => {
  return function curried(...args) {
    if (args.length >= fn.length) return fn(...args);
    return (...more) => curried(...args, ...more);
  };
};

// Pipe — left-to-right (reads naturally)
const pipe = (...fns) => (x) => fns.reduce((acc, fn) => fn(acc), x);

const double = x => x * 2;
const addOne = x => x + 1;
const square = x => x * x;

const transform = pipe(double, addOne, square);
transform(3); // double(3)=6 → addOne(6)=7 → square(7)=49

// Real-world: user data pipeline
const processUser = pipe(
  user => ({ ...user, name:  user.name.trim() }),
  user => ({ ...user, email: user.email.toLowerCase() }),
  user => ({ ...user, createdAt: new Date() })
);

Key interview line

"Currying enables partial application — baking in some arguments to create specialised functions. Composition builds complex data pipelines from simple, single-purpose functions. I prefer pipe (left-to-right) over compose since it reads naturally. Each function in a pipeline does exactly one thing — makes unit testing trivial."

JS9

What is the difference between Deep Copy and Shallow Copy? What are the best ways to clone objects?

Shallow copy copies only top-level properties — nested objects are still shared references. Deep copy recursively copies all levels so the original and clone are completely independent.

JavaScript — shallow vs deep copy
const original = { a: 1, nested: { b: 2 } };

// Shallow — nested is a shared reference!
const shallow = { ...original };
shallow.nested.b = 999;
original.nested.b; // 999 — original mutated!

// JSON — simple deep copy, but loses functions/Date/Map/Set/undefined
const d1 = JSON.parse(JSON.stringify(original));
d1.nested.b = 999;
original.nested.b; // 2 — safe

// structuredClone — modern built-in, handles Date/Map/Set (recommended)
const d2 = structuredClone(original);

// lodash cloneDeep — most robust, handles functions too
const d3 = _.cloneDeep(original);
MethodDepthDate / Map / SetFunctions
{ ...obj }ShallowReference onlyReference only
JSON.parse/stringifyDeep❌ Loses them❌ Loses them
structuredCloneDeep✅ Yes❌ No
_.cloneDeepDeep✅ Yes✅ Yes

Key interview line

"My go-to for deep cloning in modern code is structuredClone() — it's built-in and handles Dates, Maps, and Sets. I avoid JSON.parse/stringify in production because it silently drops functions, undefined values, and Dates."

Chapter 5 — ES6+ & Advanced Features
JS10

What are the most important ES6+ features? Explain destructuring, optional chaining, nullish coalescing, and Symbols.

JavaScript — ES6+ essentials
// Destructuring
const { name, age = 25, address: { city } } = user;
const [first, , third, ...rest] = [1, 2, 3, 4, 5];
const greet = ({ name, role = 'user' }) => `Hello ${name}`;

// Spread / Rest
const merged = { ...obj1, ...obj2, extra: 'val' };
function sum(...nums) { return nums.reduce((a, b) => a + b, 0); }

// Optional chaining (?.) — safe property access
const city   = user?.address?.city;
const len    = user?.getName?.()?.length;
const first2 = arr?.[0]?.name;

// Nullish coalescing (??) vs OR (||) — critical difference!
const port  = config.port ?? 3000;  // fallback only for null/undefined
const port2 = config.port || 3000;  // fallback for 0, '', false too!

console.log(0 ?? 'default');  // 0  — 0 is not null/undefined
console.log(0 || 'default');  // 'default' — 0 is falsy!

// Symbol — unique, non-enumerable keys
const id  = Symbol('id');
const id2 = Symbol('id');
id === id2; // false — always unique

const obj = { [id]: 123, name: 'test' };
Object.keys(obj); // ['name'] — symbol hidden from iteration

// Logical assignment operators
a ||= 'default';    // a = a || 'default'
a &&= transform(a); // a = a && transform(a)
a ??= 'fallback';   // a = a ?? 'fallback'
Common gotcha

The ?? vs || distinction is a frequent bug source — ?? only falls back on null/undefined, while || falls back on any falsy value including 0 and empty string. Always prefer ?? when the fallback should only apply to absent values, not "falsy" ones.

Key interview line

"Symbols are great for private-ish keys that don't appear in Object.keys() or JSON serialization. Optional chaining combined with ?? makes working with deeply nested API responses safe without defensive if-chains."

JS11

What are Generators and Iterators? How do they work and what are their practical use cases?

An Iterator is any object with a next() method returning { value, done }. A Generator (function*) can pause and resume execution via yield — enabling lazy, on-demand evaluation.

JavaScript — generators & iterators
// Basic generator
function* count() { yield 1; yield 2; yield 3; }
const gen = count();
gen.next(); // { value: 1, done: false }
gen.next(); // { value: 2, done: false }
gen.next(); // { value: 3, done: false }
gen.next(); // { value: undefined, done: true }

// Infinite sequence — lazy, never runs out of memory
function* infiniteIds() {
  let id = 1;
  while (true) yield id++;
}
const ids = infiniteIds();
ids.next().value; // 1
ids.next().value; // 2

// Async generator — paginated API fetcher
async function* fetchPages(url) {
  let page = 1;
  while (true) {
    const data = await (await fetch(`${url}?page=${page}`)).json();
    if (!data.length) return;
    yield data;
    page++;
  }
}
for await (const page of fetchPages('/api/users')) {
  console.log('Processing:', page);
}

// Custom iterable with Symbol.iterator
const range = {
  from: 1, to: 5,
  [Symbol.iterator]() {
    let cur = this.from, last = this.to;
    return { next: () => cur <= last
      ? { value: cur++, done: false }
      : { done: true } };
  }
};
[...range]; // [1, 2, 3, 4, 5]

Key interview line

"Generators are perfect for lazy evaluation — you only compute values when asked. Key use cases: infinite sequences, paginated API fetching with async generators, and custom iterables. Redux-Saga uses generators extensively for complex async flow control."

Chapter 6 — Patterns & Performance
JS12

How does Memory Management work in JavaScript? What causes memory leaks and how do you detect and fix them?

JS uses Garbage Collection (mark-and-sweep). Objects are GC'd when no live references point to them. A memory leak = memory that is no longer needed but never released because a reference still exists somewhere.

JavaScript — common leak patterns & fixes
// ❌ Leak 1: Forgotten event listeners
btn.addEventListener('click', handler);
// ✅ Fix: always remove on cleanup
btn.removeEventListener('click', handler);

// ❌ Leak 2: Closure capturing large data unnecessarily
let bigData = new Array(1000000).fill('x');
const result = process(bigData);
bigData = null; // ✅ release reference after use

// ❌ Leak 3: Growing Map/cache never cleared
const cache = new Map();
function store(key, data) { cache.set(key, data); } // grows forever!
// ✅ Fix: use WeakMap — entries auto-GC'd when key is GC'd
const cache = new WeakMap();

// ❌ Leak 4: setInterval never cleared
// ✅ Fix: clear on component unmount (React)
useEffect(() => {
  const id = setInterval(() => poll(), 1000);
  return () => clearInterval(id);
}, []);
  • Detection: Chrome DevTools Memory tab — take heap snapshots over time; compare to see what's accumulating
  • WeakMap / WeakRef: use for caches tied to object lifetimes — entries clear automatically when the key is GC'd
  • Detached DOM nodes: remove all JS references to DOM elements after removing them from the tree

Key interview line

"The most common leaks in production: forgotten event listeners in SPAs, setInterval never cleared on unmount, and closures accidentally capturing large objects. I diagnose with Chrome DevTools heap snapshots — comparing two snapshots shows exactly what's accumulating."

JS13

What JavaScript Design Patterns do you use regularly? Explain Module, Observer, Proxy, and Decorator.

JavaScript — design patterns
// 1. Module Pattern — private state via IIFE
const UserModule = (() => {
  let _users = [];
  return {
    add:    (u) => _users.push(u),
    getAll: ()  => [..._users],
    count:  ()  => _users.length,
  };
})();
UserModule._users; // undefined — truly private

// 2. Observer Pattern — decoupled pub/sub
class EventEmitter {
  constructor() { this.events = {}; }
  on(event, fn)  { (this.events[event] ??= []).push(fn); }
  off(event, fn) { this.events[event] = this.events[event]?.filter(f => f !== fn); }
  emit(event, ...args) { this.events[event]?.forEach(fn => fn(...args)); }
}

// 3. Proxy Pattern — intercept object operations (Vue 3's reactivity)
const user = new Proxy({}, {
  set(target, key, value) {
    if (key === 'age' && typeof value !== 'number')
      throw new TypeError('Age must be a number');
    target[key] = value;
    return true;
  }
});
user.age = 'old'; // TypeError
user.age = 30;    // OK

// 4. Decorator Pattern — add cross-cutting concerns without modifying original
function withCache(fn) {
  const cache = new Map();
  return (...args) => {
    const key = JSON.stringify(args);
    if (cache.has(key)) return cache.get(key);
    const result = fn(...args);
    cache.set(key, result);
    return result;
  };
}
const add = (a, b) => a + b;
const cachedAdd = withCache(add);

Key interview line

"Proxy is underused but incredibly powerful — Vue 3's reactivity system is built entirely on it. Decorator is how I add cross-cutting concerns like caching and auth without modifying original functions — same concept as NestJS interceptors. Observer is foundational to Redux, RxJS, and EventEmitter."

JS14

What is Event Delegation? How does event bubbling and capturing work, and why is delegation more performant?

Bubbling — event fires on target then travels UP through ancestors to document. Capturing — event travels DOWN from document to target first. Event Delegation — attach ONE listener on a parent to handle events from all children via bubbling.

JavaScript — event delegation
// Propagation phases
// Capture: document → html → body → parent → target
// Bubble:  target → parent → body → html → document

element.addEventListener('click', handler, true);  // capturing
element.addEventListener('click', handler, false); // bubbling (default)

e.stopPropagation();          // stop bubbling up
e.preventDefault();           // prevent browser default

// ❌ Bad — 1000 items = 1000 listeners, breaks for dynamic items
document.querySelectorAll('li').forEach(li =>
  li.addEventListener('click', handleClick)
);

// ✅ Good — ONE listener on parent handles ALL children
document.getElementById('list').addEventListener('click', (e) => {
  const li = e.target.closest('li'); // traverse up to intended target
  if (!li) return;

  const action = li.dataset.action;
  if (action === 'delete') deleteItem(li.dataset.id);
  if (action === 'edit')   editItem(li.dataset.id);
});
// Also works for dynamically added items — no re-attaching needed!

Key interview line

"Instead of attaching listeners to every child, attach one to the parent and use event.target to identify what was clicked. Critical for dynamic lists where items are added/removed. I use closest() to safely traverse up to the intended target. React's synthetic event system uses delegation internally for the same reason."

JS15

How do you optimise JavaScript performance? Explain debounce, throttle, lazy loading, and Web Workers.

JavaScript — debounce & throttle
// Debounce — execute only after user stops for N ms
function debounce(fn, delay) {
  let timer;
  return (...args) => {
    clearTimeout(timer);
    timer = setTimeout(() => fn(...args), delay);
  };
}
input.addEventListener('input', debounce(search, 300));

// Throttle — execute at most once per N ms
function throttle(fn, limit) {
  let inThrottle;
  return (...args) => {
    if (!inThrottle) {
      fn(...args);
      inThrottle = true;
      setTimeout(() => (inThrottle = false), limit);
    }
  };
}
window.addEventListener('scroll', throttle(handleScroll, 100));
FeatureDebounceThrottle
Fires whenAfter user stops for N msAt most once per N ms
Use caseSearch input, resize end, form validationScroll handler, mouse move, button spam
JavaScript — Web Workers & lazy loading
// Web Workers — offload CPU-heavy work off the main thread
const worker = new Worker('./worker.js');
worker.postMessage({ data: hugeArray });
worker.onmessage = (e) => console.log('Result:', e.data);

// worker.js — runs in separate thread, no DOM access
self.onmessage = ({ data }) => {
  self.postMessage(heavyComputation(data.data));
};

// Dynamic import — code splitting / lazy loading
const loadChart = async () => {
  const { Chart } = await import('./ChartComponent.js');
  return Chart;
};

// IntersectionObserver — lazy-load when element enters viewport
const observer = new IntersectionObserver((entries) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      loadComponent(entry.target);
      observer.unobserve(entry.target);
    }
  });
}, { threshold: 0.1 });

// requestAnimationFrame — smooth animations (syncs with 60fps repaint)
function loop() { animate(); requestAnimationFrame(loop); }
requestAnimationFrame(loop);

Key interview line

"Debounce waits for the user to stop (search box); throttle fires at a regular rate (scroll handler). For CPU-heavy tasks I offload to Web Workers so the UI thread stays responsive. For animations, requestAnimationFrame is mandatory — it syncs with the browser's 60fps repaint cycle."

How does Type Coercion work in JavaScript? Explain == vs ===, typeof gotchas, and implicit conversion rules.

Type coercion = JavaScript automatically converts values to a common type during comparisons or operations. == triggers coercion; === (strict equality) does not — it checks value AND type.

// typeof gotchas
typeof null;        // 'object'  — historic bug, never fixed
typeof [];          // 'object'  — use Array.isArray() instead
typeof function(){}; // 'function' — functions are objects but get own type
typeof NaN;         // 'number'  — NaN is technically of type number!
typeof undefined;   // 'undefined'

// Reliable type checks
Array.isArray([]);                       // true
Object.prototype.toString.call(null);   // '[object Null]'
Object.prototype.toString.call([]);     // '[object Array]'
Number.isNaN(NaN);                       // true  — use this, NOT global isNaN()
Number.isNaN('hello');                  // false — global isNaN('hello') returns true!

// Implicit coercion in operations
1 + '2';          // '12'   — + with string triggers concatenation
'5' - 2;          // 3      — -, *, / always coerce to number
true + true;       // 2      — booleans coerce to 0/1
[] + [];           // ''     — both coerce to ''
[] + {};           // '[object Object]'
{} + [];           // 0      — {} parsed as block, +[] = 0

// Symbol.toPrimitive — control custom coercion
const obj = {
  [Symbol.toPrimitive](hint) {
    if (hint === 'number')  return 42;
    if (hint === 'string')  return 'hello';
    return true; // default
  }
};
+obj;        // 42
`${obj}`;    // 'hello'
obj + '';   // 'true'

// Object.is — stricter than ===, handles edge cases
Object.is(NaN, NaN);   // true  — unlike ===
Object.is(0, -0);     // false — unlike ===
0 === -0;            // true  — === fails to distinguish
Avoid at all costs

Never use == in production code — the coercion rules are too complex to reason about reliably. Always use ===. The only arguably acceptable use is x == null which checks for both null and undefined simultaneously.

Key interview line

"I always use === — the == coercion rules require memorising the Abstract Equality Comparison algorithm, and edge cases like [] == ![] being true make it unreliable. The three gotchas I cite are: typeof null === 'object', NaN !== NaN (use Number.isNaN), and + with a string triggers concatenation while - always coerces to number."

Strict mode ('use strict') opts into a restricted variant of JavaScript that eliminates silent errors, fixes this binding ambiguity, and prohibits syntax reserved for future versions. ES6 modules and classes are strict by default.

"ES6 modules and classes are strict by default, so in modern codebases you rarely write 'use strict' manually. How do you handle errors in JavaScript? Explain try/catch/finally, async error handling, custom errors, and unhandled rejections.

JavaScript has synchronous errors (thrown exceptions) and asynchronous errors (rejected Promises). They require different handling strategies. Unhandled Promise rejections crash Node.js processes in production.

JavaScript — error handling patterns
// try/catch/finally — finally ALWAYS runs
try {
  const data = JSON.parse(badJson);
} catch (err) {
  console.error(err.name, err.message); // SyntaxError, Unexpected token...
} finally {
  cleanup(); // runs whether or not error was thrown
}

// Custom error classes — preserve stack trace, add context
class AppError extends Error {
  constructor(message, statusCode) {
    super(message);
    this.name = 'AppError';
    this.statusCode = statusCode;
    Error.captureStackTrace(this, this.constructor); // cleaner stack
  }
}
throw new AppError('Not found', 404);

// Async error handling
// ❌ Unhandled rejection — crashes Node in production
async function fetchData() {
  const res = await fetch('/api'); // throws if network fails — uncaught!
}

// ✅ Caught with try/catch
async function fetchData() {
  try {
    const res  = await fetch('/api');
    const data = await res.json();
    return data;
  } catch (err) {
    // catches both fetch failure AND json parse error
    throw new AppError(err.message, 500);
  }
}

// ✅ Promise .catch() — equivalent for promise chains
fetch('/api').then(r => r.json()).catch(err => console.error(err));

// Global safety net — catch anything that slips through
// Browser
window.addEventListener('unhandledrejection', (e) => {
  console.error('Unhandled rejection:', e.reason);
  e.preventDefault(); // suppress console error in prod
});
// Node.js
process.on('unhandledRejection', (reason, promise) => {
  console.error('Unhandled rejection', reason);
  process.exit(1); // fail fast in production
});

// Error types worth knowing
new TypeError('wrong type');       // wrong type passed
new RangeError('out of range');    // value out of allowed range
new ReferenceError('not defined'); // accessing undeclared var
new SyntaxError('bad syntax');     // invalid code
Common mistake

Only Error objects (and subclasses) have a .stack property. Throwing primitives — throw 'something went wrong' — loses stack trace context entirely. Always throw new Error(...).

Key interview line

"I always use custom error classes that extend Error — they carry statusCode, isOperational, and other context that a generic Error can't convey. In async code, a single try/catch wrapping multiple await calls catches all rejections in that block. In Node, I always register a process.on('unhandledRejection') handler that logs and exits — silent failures are worse than crashes."

What is Object.defineProperty()? Explain property descriptors, getters/setters, and how they relate to Proxy.

Every object property has a descriptor — metadata controlling whether it can be written, enumerated, or reconfigured. Object.defineProperty() gives you fine-grained control over this, and getters/setters add computed behaviour directly on an object.

Descriptor flagDefault (defineProperty)Meaning
writablefalseCan the value be changed with =?
enumerablefalseDoes it appear in for...in / Object.keys()?
configurablefalseCan the descriptor itself be changed or deleted?
JavaScript — defineProperty & getters/setters
// Read-only, non-enumerable property
const obj = {};
Object.defineProperty(obj, 'id', {
  value: 42,
  writable:     false,
  enumerable:   false,
  configurable: false
});
obj.id = 99;          // silently fails (TypeError in strict)
Object.keys(obj);     // []  — not enumerable, hidden
obj.id;               // 42  — still readable

// Getter / setter — computed property
const person = { _name: 'Alice' };
Object.defineProperty(person, 'name', {
  get() { return this._name.toUpperCase(); },
  set(v) { this._name = v.trim(); },
  enumerable: true,
  configurable: true
});
person.name;          // 'ALICE'
person.name = '  Bob  ';
person._name;         // 'Bob'

// Shorthand get/set in object literal (same thing)
const circle = {
  _r: 5,
  get area()  { return Math.PI * this._r ** 2; },
  set radius(v) { this._r = v; }
};

// defineProperty vs Proxy
// defineProperty — static, per-property, defined at declaration time
// Proxy — dynamic, intercepts ALL operations, even on not-yet-existing properties

// Object.getOwnPropertyDescriptor — inspect any property
Object.getOwnPropertyDescriptor([], 'length');
// { value: 0, writable: true, enumerable: false, configurable: false }

// Inspect all descriptors at once
Object.getOwnPropertyDescriptors(person);

Key interview line

"Object.defineProperty is how the JS engine itself defines built-in properties — Array.length is non-enumerable, for example. Getters/setters let you add validation or computation without changing the consumer's API. The key distinction vs Proxy is scope: defineProperty targets a specific known property; Proxy intercepts all operations dynamically, including property creation."

What is Immutability in JavaScript? Explain const vs Object.freeze() vs deep freeze, and why it matters.

const makes the binding immutable — you can't reassign the variable. Object.freeze() makes the object immutable — you can't add, remove, or change properties. Neither is recursive by default.

TechniquePrevents reassignmentPrevents mutationDeep?
const✅ Yes❌ No—
Object.freeze()—✅ Shallow❌ No
Object.seal()—Partial (no add/delete, can change)❌ No
Deep freeze (recursive)—✅ All levels✅ Yes
JavaScript — immutability patterns
// const — only binding is locked
const user = { name: 'Alice' };
user.name = 'Bob';  // ✅ mutation works fine
user = {};          // ❌ TypeError — can't reassign const

// Object.freeze — shallow, properties can't change
const config = Object.freeze({ db: { host: 'localhost' } });
config.db.host = 'prod'; // silently mutates! freeze is shallow
config.port = 3000;     // silently ignored (TypeError in strict)

// Object.seal — can change existing, can't add or delete
const sealed = Object.seal({ x: 1 });
sealed.x = 2;    // ✅ OK
sealed.y = 3;    // ❌ silently ignored
delete sealed.x;  // ❌ silently ignored

// Deep freeze — recursive
function deepFreeze(obj) {
  Object.getOwnPropertyNames(obj).forEach(key => {
    const val = obj[key];
    if (typeof val === 'object' && val !== null) deepFreeze(val);
  });
  return Object.freeze(obj);
}

// Immutable update pattern (spread) — preferred in React/Redux
const nextState = { ...state, count: state.count + 1 };       // shallow
const deepUpdate = { ...state, user: { ...state.user, name: 'Bob' } }; // nested

// Check if frozen/sealed
Object.isFrozen(config);  // true
Object.isSealed(sealed);  // true

Key interview line

"const and Object.freeze() are often confused — const prevents rebinding, freeze prevents mutation. In React/Redux I use the spread pattern for immutable updates rather than freeze, because immutable updates produce new references which triggers re-render detection efficiently. Object.freeze() is useful for config objects and constants that should never change at runtime."

What is the difference between CommonJS (require) and ES Modules (import/export)? How do circular dependencies behave in each?

CommonJS is Node.js's original module system — synchronous, runtime-evaluated. ES Modules (ESM) are the standard — statically analysable, async-capable, and the basis for tree shaking. They are NOT interchangeable.

AspectCommonJS (require)ES Modules (import)
LoadingSynchronous, runtimeAsynchronous, parsed statically
ExportsCopy of value at require timeLive binding — reflects later changes
Top-level await❌ Not supported✅ Supported
Tree shaking❌ Not possible✅ Bundlers can eliminate unused exports
Dynamic importrequire() anywhereimport() returns a Promise
Default in Node.js files.mjs or "type":"module" in package.json
JavaScript — CJS vs ESM behaviour
// ── CommonJS ──────────────────────────────────────
// counter.js
let count = 0;
module.exports = { count, increment: () => count++ };

// main.js
const { count, increment } = require('./counter');
increment();
console.log(count); // 0 — CJS exports a COPY at require time

// ── ES Modules ────────────────────────────────────
// counter.mjs
export let count = 0;
export const increment = () => count++;

// main.mjs
import { count, increment } from './counter.mjs';
increment();
console.log(count); // 1 — ESM exports a LIVE BINDING

// Circular dependency behaviour
// CJS — returns whatever is exported SO FAR (incomplete object, often {})
// ESM — live bindings handle circulars better; binding exists but may be
//       undefined until the depended module fully evaluates

// Dynamic import — lazy load in both ESM and CJS contexts
const { default: lodash } = await import('lodash'); // works in ESM
const lodash2 = require('lodash');                  // CJS, synchronous

// Named vs default exports — best practice
export default function main() {}      // one per module
export const helper = () => {};         // multiple named exports
import main, { helper } from './mod.js';
Interop caveat

You can require() an ESM module from CJS in Node 22+ using require(esm) experimentally, but in general ESM cannot be require'd — you'll get ERR_REQUIRE_ESM. CJS can always be imported from ESM using import, but only the default export is available.

Key interview line

"The most important practical difference: CJS exports a value copy at require time — mutations in the original module are invisible to importers. ESM exports live bindings — changes are reflected. This is why tree shaking only works with ESM — bundlers can statically analyse which exports are used and eliminate the rest at build time."
