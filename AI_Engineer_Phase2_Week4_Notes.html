01

Zero-Shot Prompting

You describe the task. No examples. The model uses its training distribution to generalize. Zero-shot is the baseline — test it first before adding complexity.

Python · Anthropic SDK
import anthropic
client = anthropic.Anthropic()

# Zero-shot: describe the task, no examples
response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=64,
    messages=[{
        "role": "user",
        "content": (
            "Classify the sentiment as POSITIVE, NEGATIVE, or NEUTRAL.\n\n"
            "Review: The battery life is amazing but the screen is dim.\n\n"
            "Sentiment:"
        )
    }]
)
# → MIXED  (inconsistent — no examples defined the edge case)
print(response.content[0].text)
When to use zero-shot

Tasks the model handles natively from training: translation, summarization, simple Q&A, basic classification. Always test zero-shot first — it's the cheapest option and often sufficient. Only add few-shot examples when zero-shot is measurably inconsistent.

02

Few-Shot Prompting

You provide 2–5 labeled examples before the actual input. This conditions the model's output distribution toward your pattern — not by training, but by shaping the context the next token is predicted from.

Python — Few-shot pattern
few_shot = """Classify sentiment as POSITIVE, NEGATIVE, or NEUTRAL.
Mixed opinions (good + bad) count as NEUTRAL.

Review: "Best laptop I've ever owned. Fast, light, gorgeous screen."
Sentiment: POSITIVE

Review: "Stopped working after 2 weeks. Support was useless."
Sentiment: NEGATIVE

Review: "Great camera, terrible battery. Evens out."
Sentiment: NEUTRAL

Review: "The battery life is amazing but the screen is dim."
Sentiment:"""

response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=10,
    messages=[{"role": "user", "content": few_shot}]
)
# → NEUTRAL  (consistent now — examples defined the edge case rule)

Few-shot best practices

Python — What makes examples effective
# 1. DIVERSITY — cover edge cases, not just easy cases
# Bad:  3 clearly POSITIVE reviews as examples
# Good: 1 POSITIVE, 1 NEGATIVE, 1 NEUTRAL (the hard mixed case)

# 2. ORDER MATTERS — last example slightly biases toward its label
# If last example is POSITIVE, model leans POSITIVE on ambiguous inputs
# Fix: shuffle example order per request in production
import random
examples = [pos_example, neg_example, neutral_example]
random.shuffle(examples)

# 3. FORMAT CONSISTENCY — examples define the output format exactly
# If examples output "POSITIVE" → model outputs "POSITIVE" (not "positive")
# Be deliberate: capitalization, punctuation, spacing all transfer

# 4. BALANCE — roughly equal examples per class
# 3 POSITIVE + 1 NEGATIVE → biased output

# 5. HARD CASES — your examples should include your failure modes
# Run zero-shot on 20 inputs, collect the wrong ones, add them as examples
# This is the most efficient use of few-shot slots

🧠 Why examples actually work

Each example is tokens in the context. The model's next-token prediction is conditioned on all preceding tokens — so your examples literally shift the probability distribution at inference time. The model is not "learning" in any training sense. It's pattern-matching: "given that the last 3 examples all followed format X, the next completion should also follow format X." This is why format consistency in examples matters so much.

03

Chain-of-Thought (CoT)

You tell the model to reason step-by-step before committing to a final answer. This dramatically improves performance on multi-step reasoning tasks. The intermediate reasoning tokens serve as a scratchpad — they condition better final-answer tokens.

Python — Zero-shot CoT vs no CoT
# WITHOUT CoT — model often wrong on multi-step arithmetic
no_cot = (
    "A store has 5 apples. They sell 2, receive a shipment of 3× what "
    "they now have, then sell half. How many remain?\n\nAnswer:"
)
# → "6"  (often wrong, inconsistent)

# WITH Zero-shot CoT — just add "Think step by step."
with_cot = (
    "A store has 5 apples. They sell 2, receive a shipment of 3× what "
    "they now have, then sell half. How many remain?\n\n"
    "Think step by step before giving your final answer."
)
# Model reasons: Start=5 → sell 2 → 3 → shipment: 3×3=9, total=12 → half=6
# → "6"  ✓ (correct AND consistent)

# Classic CoT failure: the bat and ball problem
trick = "A bat and ball cost $1.10 total. The bat costs $1 more than the ball. How much is the ball? Think step by step."
# Without CoT → "10 cents" (the intuitive wrong answer)
# With CoT    → "If ball=x, bat=x+1, so 2x+1=1.10, x=0.05 → 5 cents" ✓

Structured CoT — controlled reasoning format

Python — XML-structured reasoning
structured_cot = """Analyze this support ticket and determine priority.

TICKET: "I've been charged twice and my account shows suspended."

Work through your analysis:

Issue type: [billing/technical/account/other]
Business impact: [low/medium/high/critical]
Customer emotion: [calm/frustrated/angry/urgent]
Resolution complexity: [simple/moderate/complex]


[P1/P2/P3/P4]

[1–2 sentence explanation]"""

# Why this works better than free-form CoT:
# 1. Forces the model to consider specific dimensions before deciding
# 2. The XML tags are parseable — you can extract each field
# 3. The intermediate analysis tokens improve the final answer quality
# 4. Consistent structure makes it easy to evaluate and test
TechniqueTrigger phraseBest forCost
Zero-shot CoT"Think step by step."Math, logic, multi-step+30–100 output tokens
Few-shot CoTExamples with reasoning shownDomain-specific reasoning+300–500 input tokens
Structured CoTXML tags for reasoning stepsComplex decisions, evaluation+100–200 tokens, parseable
Self-consistencyRun N times, take majority voteHigh-stakes single answersN× total cost
Why CoT works — the mechanistic explanation

The model generates reasoning tokens that become part of the context for subsequent tokens. The final answer token is predicted from: [question tokens] + [reasoning tokens]. More reasoning tokens = richer context = higher probability of the correct answer. You're giving the model more surface area to work with before committing.

System Prompt Design

The system prompt is your most powerful lever. It sets identity, constraints, output format, and behavioral guardrails — applied once before the user says anything. A well-designed system prompt makes every user turn more predictable and reliable.

❌ Vague — don't do this

"You are a helpful assistant that answers questions about our product. Be friendly and professional."

✓ Specific — do this

"You are a customer support agent for Acme SaaS. Answer questions about features, pricing, and integrations only. Never fabricate features..."

The 5-component system prompt template

## [1] ROLE & IDENTITY You are a {seniority} {job_title} at {specific_context}. Your primary purpose is {one sentence goal}. ## [2] CAPABILITIES & SCOPE You CAN: {list what to do} You CANNOT: {explicit hard limits — more important than capabilities} ## [3] RULES & GUARDRAILS - When unsure: {exactly what to say — e.g. "I don't have that info"} - When out of scope: {redirect message verbatim} - Edge case rule: {specific behavioral rule} ## [4] OUTPUT FORMAT Format: {JSON / XML / markdown / plain text} Length: {hard cap — e.g. "under 150 words unless walkthrough needed"} Structure: {example of exact output shape} ## [5] TONE {3 adjectives}. Example voice: "[single sentence example]"
Python — Production system prompt example
system = """You are a customer support agent for Acme SaaS,
a B2B project management tool.

## Role
Answer questions about Acme's features, pricing, and integrations.
Help users solve problems within the product.

## Rules
- Only answer questions about Acme. For off-topic: "That's outside
  my expertise — I can help with [relevant topic] instead."
- Never make up features. If unsure: "I don't have that information.
  Type 'human' to connect with our team."
- Keep responses under 150 words unless a step-by-step walkthrough
  is required.
- Always end with: "Was this helpful? Type 'human' for a live agent."

## Output format
- Use numbered steps for instructions
- Bold key terms on first mention
- For pricing: always link https://acme.io/pricing

## Tone
Professional, warm, direct. Not corporate. Not overly casual.
Example: "Hey! I looked into that — here's what's happening..." """

System prompt testing checklist

Python — 8 required test cases
test_cases = {
    "happy_path":      "Normal question the prompt is designed for",
    "edge_case":       "Question at the boundary of your defined scope",
    "out_of_scope":    "Question completely outside your domain",
    "adversarial":    "'Ignore previous instructions and...' attack",
    "ambiguous":      "Question with multiple valid interpretations",
    "empty_input":    "",
    "very_long_input": "A" * 5000,
    "sensitive_topic": "Something that might trigger model refusals",
}

# Run ALL 8 every time you change the system prompt.
# Treat it like a unit test suite — a change that fixes one case
# might silently break another.
Key principle — constraints > instructions

Negative constraints are more valuable than positive instructions. "Never make up pricing" prevents a specific catastrophic failure mode. "Be helpful" is too vague to reliably shape behavior. For every capability you add, write one explicit constraint that prevents the most likely bad version of it.

XML Structured Outputs

Claude was trained on XML-structured data and handles it exceptionally well. Use XML tags to structure both your prompts and the outputs you request. It is more robust than JSON for cases where the model might insert extra prose.

Python — XML extraction + parsing
import xml.etree.ElementTree as ET
import re

extraction_prompt = """Extract structured data from the job posting below.


{job_text}


Return your extraction in this EXACT XML format and nothing else:


  [job title]
  [company name]
  [city, country or "Remote"]
  [number or null]
  [number or null]
  
    [skill 1]
    [skill 2]
  
  [number or null]
  [true/false]
"""

def parse_xml_response(response_text: str) -> dict:
    # Extract just the XML block — model might add preamble
    match = re.search(r'<extraction>.*?</extraction>', response_text, re.DOTALL)
    if not match:
        raise ValueError("No <extraction> block found in response")

    root   = ET.fromstring(match.group())
    skills = [s.text for s in root.findall('.//skill')]

    return {
        "title":            root.findtext('title'),
        "company":          root.findtext('company'),
        "location":         root.findtext('location'),
        "salary_min":       root.findtext('salary_min'),
        "salary_max":       root.findtext('salary_max'),
        "required_skills":  skills,
        "experience_years": root.findtext('experience_years'),
        "remote":           root.findtext('remote') == 'true',
    }

# Why XML beats JSON for extraction prompts:
# - Claude handles nested XML reliably out of the box
# - Easier to extract from response that has extra prose around it
# - Lists (repeated <skill> tags) are trivial vs JSON arrays
# - Model is less likely to break XML structure mid-generation

JSON Outputs + Pydantic Validation

For programmatic use, JSON + Pydantic is the production standard. Pydantic validates types, coerces values, and gives you precise error messages you can feed back into a retry.

Python · Pydantic v2 — Schema + extraction
from pydantic import BaseModel, Field, field_validator
from typing import Optional, List
import json, re

class JobPosting(BaseModel):
    title:            str
    company:          str
    location:         str
    salary_min:       Optional[int]   = None
    salary_max:       Optional[int]   = None
    required_skills:  List[str]       = Field(default_factory=list)
    experience_years: Optional[int]   = None
    remote:           bool             = False

    @field_validator('salary_min', 'salary_max', mode='before')
    @classmethod
    def parse_salary(cls, v):
        if v in (None, 'null', 'None', ''):
            return None
        return int(str(v).replace(',', '').replace('$', '').replace('k','000'))

def extract_job(text: str) -> JobPosting:
    prompt = f"""Extract job details from this posting.

<posting>
{text}
</posting>

Return ONLY a JSON object matching this schema exactly:
{JobPosting.model_json_schema()}

Rules:
- JSON only. No explanation. No markdown fences.
- Use null (not "N/A" or "") for missing optional fields.
- required_skills must be a JSON array, even if empty: []"""

    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=512,
        temperature=0.0,   # deterministic for extraction
        messages=[{"role": "user", "content": prompt}]
    )

    raw = response.content[0].text.strip()
    raw = re.sub(r'^```json\s*|\s*```$', '', raw, flags=re.MULTILINE)

    return JobPosting(**json.loads(raw))  # Pydantic validates + coerces

Retry Logic — Production-Grade Extraction

Models occasionally produce malformed JSON, miss required fields, or use wrong types. The fix: catch the validation error and feed it back into the next attempt. The model almost always corrects itself when shown exactly what went wrong.

Python — Generic retry wrapper for any Pydantic schema
from typing import TypeVar, Type
import time
T = TypeVar('T', bound=BaseModel)

def extract_with_retry(
    text:        str,
    schema:      Type[T],
    max_retries: int = 3,
) -> T:
    """
    Attempt extraction, catch validation errors,
    inject the error into the next prompt, retry.
    """
    last_error  = None
    raw_output  = None

    for attempt in range(max_retries):
        # On retries: include previous failure so model can self-correct
        error_ctx = ""
        if last_error and raw_output:
            error_ctx = f"""

Your previous attempt failed validation with this error:
<previous_output>{raw_output}</previous_output>
<error>{last_error}</error>

Fix the error and try again. JSON only."""

        prompt = f"""Extract structured data from the text below.
Return ONLY valid JSON matching this schema:
{schema.model_json_schema()}

<text>{text}</text>
{error_ctx}"""

        response = client.messages.create(
            model="claude-sonnet-4-20250514",
            max_tokens=1024,
            temperature=0.0 if attempt == 0 else 0.3,
            messages=[{"role": "user", "content": prompt}]
        )

        raw_output = response.content[0].text.strip()
        raw_output = re.sub(r'^```json\s*|\s*```$', '', raw_output, flags=re.MULTILINE)

        try:
            data = json.loads(raw_output)
            return schema(**data)        # success — return validated model
        except Exception as e:
            last_error = str(e)
            print(f"  Attempt {attempt+1} failed: {e}")
            time.sleep(0.5 * (attempt + 1))  # exponential backoff

    raise RuntimeError(f"Failed after {max_retries} attempts: {last_error}")

# Usage — works for any Pydantic model:
job     = extract_with_retry(job_text,   JobPosting)
receipt = extract_with_retry(receipt_txt, Receipt)
article = extract_with_retry(news_text,   NewsArticle)
Temperature on retry

Use temperature=0.0 on the first attempt for maximum determinism. On retries, bump to 0.3 — a small amount of randomness helps the model escape whatever caused the first failure. If attempt 1 produced malformed JSON because of some local minimum in the distribution, a slightly different sampling path on attempt 2 often avoids it.

The Prefilling Trick

Prefilling means seeding the assistant turn with a partial response. Claude continues from exactly that string — no preamble, no "Certainly! Here's...", no hedging. It's the most underused power technique in the Anthropic SDK.

Python · Anthropic SDK — Prefilling patterns
# ── WITHOUT prefilling — unpredictable opener ─────────────────
response = client.messages.create(
    model="claude-sonnet-4-20250514", max_tokens=256,
    messages=[{"role": "user",
               "content": "Write a function to validate email addresses."}]
)
# → "Sure! Here's a Python function that validates email addresses:\n```python\n..."
#   ↑ preamble is noise — hard to parse, clutters downstream pipelines

# ── WITH prefilling — clean output, no preamble ───────────────
response = client.messages.create(
    model="claude-sonnet-4-20250514", max_tokens=256,
    messages=[
        {"role": "user",
         "content": "Write a function to validate email addresses."},
        {"role": "assistant",
         "content": "```python\nimport re\n\ndef validate_email(email: str) -> bool:"}
    ]
)
# → continues: "    pattern = r'^[...]+@[...]+\.[a-zA-Z]{2,}$'\n    ..."
# Zero preamble. Clean code. Directly parseable.

# ── Force JSON output ─────────────────────────────────────────
response = client.messages.create(
    model="claude-sonnet-4-20250514", max_tokens=512, temperature=0.0,
    messages=[
        {"role": "user",
         "content": f"Extract key facts from:\n\n{article_text}"},
        {"role": "assistant",
         "content": '{"facts": ['}  # seeds valid JSON start
    ]
)
# Response continues: '"Fact 1", "Fact 2"], "summary": "..."}'
# Add the seed back: full_json = '{"facts": [' + response.content[0].text

# ── Force XML ─────────────────────────────────────────────────
response = client.messages.create(
    model="claude-sonnet-4-20250514", max_tokens=256,
    messages=[
        {"role": "user", "content": f"Analyze sentiment:\n\n{review}"},
        {"role": "assistant", "content": "<analysis>"}  # guaranteed XML wrapper
    ]
)

Hallucination — Causes & Mitigation

Hallucination is the model generating confident, plausible-sounding, factually wrong output. Understanding why it happens leads directly to the fixes.

Why hallucination happens

The model predicts the most probable next token given context.
"The capital of Australia is ___" → "Sydney" is more probable than "Canberra"
because Sydney appears more often in internet text.

There is no "I don't know" signal — the model was trained to complete text,
not to flag uncertainty. Uncertainty must be engineered in explicitly.

The 5 mitigation strategies

Python — All 5 hallucination mitigation patterns
# ── 1. GROUNDING — give the facts explicitly ──────────────────
# Never ask "what is the refund policy?" — the model confabulates
# DO: paste the actual policy, ask about it
grounded = """Answer using ONLY the information in the document below.
If the answer is not in the document, say EXACTLY:
"This information is not in the provided document."
Do NOT use outside knowledge.

<document>{doc}</document>

Question: {question}"""

# ── 2. CITATIONS — force the model to quote sources ───────────
citation = """Answer the question. After each factual claim, add:
[source: "exact quote from document"].
Only make claims you can directly cite from the document.

<document>{doc}</document>

Question: {q}"""

# ── 3. CONFIDENCE ELICITATION ─────────────────────────────────
confidence = """Answer the following question.
After your answer write:
CONFIDENCE: [HIGH / MEDIUM / LOW]
REASON: [one sentence — why you are or aren't confident]

Question: {q}"""

# ── 4. CHAIN-OF-THOUGHT BEFORE COMMITTING ────────────────────
verify_cot = """Before answering, reason through what you know and don't know.

<reasoning>
What I know for certain:
What I'm less sure about:
Where I might be wrong:
</reasoning>

<answer>[answer here]</answer>

Question: {q}"""

# ── 5. VERIFICATION LOOP — extract then verify ───────────────
def verified_extract(text: str, question: str) -> dict:
    # Step 1: extract an answer
    answer = client.messages.create(
        model="claude-sonnet-4-20250514", max_tokens=256,
        messages=[{"role": "user",
                   "content": f"From this text:\n{text}\n\nAnswer: {question}"}]
    ).content[0].text

    # Step 2: verify against source
    ver = client.messages.create(
        model="claude-sonnet-4-20250514", max_tokens=128, temperature=0.0,
        messages=[{"role": "user",
                   "content": f"""Is this answer supported by the source?
Source: {text}
Answer: {answer}
JSON: {{"supported": true/false, "evidence": "direct quote or null"}}"""}]
    ).content[0].text

    return {"answer": answer, "verification": json.loads(ver)}

Role Prompting & Persona Design

Assigning a specific, detailed role shifts the entire response distribution — vocabulary, depth, format, tone, and the examples the model reaches for all change. Vague roles ("expert") produce vague shifts. Specific roles ("principal engineer who debugged production incidents at a bank") produce specific, useful shifts.

❌ Vague role — weak effect

"You are an expert. Explain database indexing."

✓ Specific role — strong effect

"You are a senior DB engineer who debugged slow queries on tables with 500M+ rows. Explain indexing using real production examples."

Python — Role design formula + examples
# Role design formula:
# "You are a {seniority} {job_title} {at/with} {specific context}.
#  You have {N} years of experience in {domain}.
#  You {distinctive_behavior_1}.
#  You {distinctive_behavior_2}.
#  You never/always {hard constraint}."

roles = {
    "code_reviewer": """You are a principal engineer reviewing code
for a bank's trading system. You have 12 years in high-stakes
financial software. You flag security issues, performance
bottlenecks, and maintainability problems. You cite specific
failure modes from real production incidents. You never accept
'good enough' — every review is thorough.""",

    "data_analyst": """You are a senior data analyst at a product
analytics company. You think in terms of statistical significance,
sample sizes, and confounds. You always ask 'what could make this
interpretation wrong?' before concluding. You present findings with
confidence intervals, not just point estimates.""",

    "adversarial_reviewer": """You are a senior engineer tasked
with finding flaws in plans. Your job is to poke holes, find edge
cases, and identify what will break in production. You are
constructively critical — you explain WHY something will fail
and roughly how likely that failure is.""",
}

# Usage in API call
response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=512,
    system=roles["adversarial_reviewer"],
    messages=[{"role": "user",
               "content": f"Review this architecture proposal:\n\n{proposal}"}]
)

Multi-persona analysis — get multiple expert views

Python — Multi-perspective analysis pattern
def multi_perspective_analysis(proposal: str) -> dict:
    """Get critique from 4 expert roles simultaneously."""
    perspectives = {
        "technical": "You are a senior software architect. Identify technical risks.",
        "business":  "You are a CFO. Identify business and financial risks.",
        "security":  "You are a CISO. Identify security and compliance risks.",
        "user":      "You are a UX researcher. Identify user experience risks.",
    }

    results = {}
    for name, role in perspectives.items():
        resp = client.messages.create(
            model="claude-sonnet-4-20250514", max_tokens=200,
            system=role,
            messages=[{"role": "user",
                       "content": f"Proposal:\n{proposal}\n\nTop 3 risks:"}]
        )
        results[name] = resp.content[0].text
    return results

# Each perspective catches different blind spots

Negative Examples in Prompts

Showing the model what NOT to produce is often more effective than telling it what to produce. Positive instructions are abstract and easy to underweight; concrete bad examples are specific patterns the model can recognize and avoid.

Python — Negative examples for summarization
# WITHOUT negative examples — model still writes vague summaries
without_neg = """Summarize the article in 2-3 sentences.
Be concise and focus on the key points."""
# Model might still output: "This article discusses various aspects..."

# WITH negative examples — much tighter constraint
with_neg = """Summarize the article in 2-3 sentences.

BAD examples (do NOT write like this):
- "This article discusses various aspects of climate policy."
  WHY BAD: too vague — what aspects? Says nothing specific.
- "The author makes several interesting points about the subject."
  WHY BAD: zero content — this could describe any article.
- "In this comprehensive analysis, the writer explores how..."
  WHY BAD: bloated opener using filler words.

GOOD example:
- "Global temperatures have risen 1.1°C since industrialization.
   New UN data shows emission cuts need to triple by 2030 to meet
   Paris targets. The EU's carbon border tax is the most significant
   policy response so far."
  WHY GOOD: specific facts, concrete numbers, no filler.

Now summarize:
{article}"""

Before/After pairs — the most powerful format

Python — Before/After rewrite prompt
rewrite_prompt = """Rewrite customer support emails to be clearer and more helpful.

BEFORE (do NOT write like this):
"Dear Customer, We have received your inquiry regarding the matter
you have contacted us about. Our team is looking into this and we
will get back to you in due course. Thank you for your patience."

AFTER (write like this):
"Hi [Name], I looked into your billing issue — you were charged
twice on March 3rd. I've issued a full refund for $29.99.
It'll appear in 3-5 business days. Let me know if you need
anything else."

Key differences:
- BEFORE: passive, vague, no timeline, no specifics, no resolution
- AFTER: addresses the actual issue, gives specific action + date,
  clear timeline, leaves door open without being clingy

Now rewrite this email:
{email}"""

Negative examples for structured outputs

Python — Preventing JSON failures with negative examples
json_strict = """Extract event details and return JSON.

WRONG outputs — never do these:
❌ Adds explanation:  "Here is the extracted JSON: {...}"
❌ Markdown fences:   ```json {...} ```
❌ Missing fields:    {"name": "..."} — all fields required
❌ Invented fields:   {"attendee_count": 500} — not in source text
❌ Wrong null syntax: {"date": "unknown"} — use null, not strings

CORRECT output — exactly this format:
{"name": "...", "date": "YYYY-MM-DD", "location": "...", "organizer": "..."}

Extract from:
{text}"""

# ── Anti-pattern list (built from real production failures) ───
anti_pattern = """Generate a product description for e-commerce.

Patterns we've measured as causing low conversion (real examples):
1. "Experience the ultimate..." — overused opener, zero credibility
2. "Perfect for anyone who wants..." — too vague to create desire
3. Starting with the brand name — leads with ego, not value
4. "This product features..." — spec dump, no benefits stated
5. Bullet lists of specs without explaining WHY each matters

Lead with the problem it solves → evidence it solves it → specs.

Product: {product_details}"""
Build your anti-pattern list from real failures

The highest-value negative examples are failures you've actually observed in production. Run your prompt on 20–50 inputs. Collect the bad outputs. Annotate each with why it's bad. Add the worst offenders back into the prompt as explicit negatives. This is prompt engineering through empirical feedback — not guessing.

Python — extractor.py (engine + test harness)
import anthropic, json, re, time
from typing import TypeVar, Type
from pydantic import BaseModel

client = anthropic.Anthropic()
T      = TypeVar('T', bound=BaseModel)

def extract(text: str, schema: Type[T], max_retries=3) -> T:
    last_error = raw = None
    for attempt in range(max_retries):
        error_ctx = ""
        if last_error:
            error_ctx = f"\n\nPrevious attempt failed:\n<prev>{raw}</prev>\n<error>{last_error}</error>\nFix it. JSON only."

        prompt = f"""Extract structured data. Return ONLY valid JSON.
Schema: {schema.model_json_schema()}
Rules: no markdown fences, no preamble, null for missing optionals.

<text>{text}</text>{error_ctx}"""

        raw = client.messages.create(
            model="claude-sonnet-4-20250514", max_tokens=1024,
            temperature=0.0 if attempt == 0 else 0.3,
            messages=[{"role": "user", "content": prompt},
                      {"role": "assistant", "content": "{"}]  # prefill
        ).content[0].text
        raw = "{" + re.sub(r'^```json\s*|\s*```$', '', raw, flags=re.M)

        try:
            return schema(**json.loads(raw))
        except Exception as e:
            last_error = str(e)
            time.sleep(0.5 * (attempt + 1))
    raise RuntimeError(f"Failed after {max_retries} attempts: {last_error}")

🧠 The mental model to carry forward

Every technique in this week is about the same thing: shaping the probability distribution over output tokens. Few-shot examples shift it by conditioning. System prompts shift it by framing. CoT shifts it by inserting reasoning tokens that condition better final tokens. Prefilling shifts it by literally forcing the first tokens. Negative examples shift it by making certain token sequences less likely. Once you see all prompting as distribution shaping, every new technique has an obvious place.
