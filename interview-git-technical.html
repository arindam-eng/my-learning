Chapter 1 — Internals
T1 Core

Git objects & SHA — how Git stores everything

Git is a content-addressable filesystem. Every piece of data is stored as an object named by the SHA-1 hash of its content (moving to SHA-256). There are four object types; together they form Git's immutable object graph.

ObjectWhat it storesPoints to
blobRaw file content (no filename, no permissions)Nothing
treeDirectory listing: names, modes, blob/tree SHAsblobs, other trees
commitAuthor, timestamp, message, parent SHAsOne tree + parent commits
tagAnnotated tag metadata: tagger, message, GPG sigAny object (usually a commit)

The .git/objects/ directory holds all objects. Loose objects are stored at objects/ab/cdef… (first 2 hex chars = directory). Git periodically packs them into .git/objects/pack/ for space efficiency.

bash — Inspecting objects
# See the type of any object
git cat-file -t HEAD              # → commit
git cat-file -t HEAD^{tree}       # → tree

# Pretty-print object content
git cat-file -p HEAD              # → tree sha, parent, author, message
git cat-file -p HEAD^{tree}       # → list of blobs and subtrees

# Hash a file manually (same as what git would store)
git hash-object README.md         # → sha1 of the blob

# Count objects and space
git count-objects -vH
Say this in the interview

"Git never stores diffs — it stores complete snapshots as an object DAG. A commit points to a tree (directory snapshot), trees point to blobs (file content) or subtrees. SHA identity means identical files across commits share the same blob. That's why Git is fast and disk-efficient."

Quick recall

blob = file content · tree = directory · commit = snapshot + metadata · tag = named pointer. SHA is computed from content, so two identical files are one blob. git cat-file -p <sha> to inspect any object.

T2

Staging area & index — the three states of a file

Every tracked file in Git lives in one of three states: modified (changed in working tree), staged (added to index, ready to commit), or committed (safely in the object database).

LocationNameCommand to move forward
Working treeModifiedgit add <file>
Index / staging areaStagedgit commit
.git/objectsCommitted—

The index (.git/index) is a binary file that maps file paths to blob SHAs and metadata. git status compares: HEAD tree vs. index (staged diff), and index vs. working tree (unstaged diff).

bash — Precision staging
# Stage only specific hunks interactively
git add -p                        # y/n/s/e per hunk

# See what's staged vs committed
git diff --staged                 # index vs HEAD
git diff                          # working tree vs index

# Unstage without losing working tree changes
git restore --staged <file>

# See index content (raw)
git ls-files --stage
Interview tip

git add -p is the answer to "how do you make atomic commits when you've mixed concerns in one file?" — it lets you stage individual hunks from a single file so each commit does exactly one thing.

Quick recall

Three areas: working tree → index (staging) → object store. git diff = unstaged. git diff --staged = staged. git add -p for surgical staging. The index is a performance cache so Git doesn't re-hash unchanged files on every commit.

Chapter 2 — Branching & History
T3 Core

merge vs rebase vs squash — when to use each

All three integrate changes from one branch into another, but they produce different history shapes and have different trade-offs around collaboration and bisectability.

StrategyHistory shapeCommit preservationBest for
mergeAdds a merge commit, preserves branch topologyAll feature commits intactLong-lived shared branches, audit trails
rebaseLinear — replays commits on top of targetCommits rewritten (new SHAs)Feature branches before PR, keeping main linear
squash mergeSingle new commit on targetFeature history discardedSmall features where history is noise
bash — All three approaches
# Merge (preserves branch topology)
git checkout main
git merge feature/login            # creates merge commit
git merge --no-ff feature/login    # force merge commit even if fast-forward possible

# Rebase (linear history)
git checkout feature/login
git rebase main                    # replay feature commits on top of main
git checkout main
git merge feature/login            # now a fast-forward, no merge commit

# Squash merge (single commit)
git checkout main
git merge --squash feature/login
git commit -m "feat: add login"    # one commit with all changes
Golden rule

Never rebase shared branches. Rebase rewrites SHAs. If teammates have pulled the old SHAs, their history diverges. Only rebase local feature branches that haven't been pushed, or use --force-with-lease on branches you exclusively own.

Quick recall

merge = preserves context, adds merge commit · rebase = linear history, rewrites SHAs, never on shared branches · squash = clean target history, loses individual feature commits. For bisectability, rebase wins. For auditing, merge wins.

T4 Core

reset vs revert vs restore — undoing things correctly

Three different commands for three different undo scenarios. Getting these wrong in a shared repo is dangerous.

CommandMoves HEAD?Changes index?Changes working tree?Safe for shared branches?
git reset --soft HEAD~1YesNoNoNo — rewrites history
git reset --mixed HEAD~1YesYes (unstages)NoNo — rewrites history
git reset --hard HEAD~1YesYesYes (discards changes)Never — destructive
git revert HEADYes (new commit)YesYesYes — safe, additive
git restore <file>NoNoYes (discards file changes)N/A (local only)
bash — Practical examples
# Undo last commit, keep changes staged (to re-commit with better message)
git reset --soft HEAD~1

# Undo last commit, keep changes but unstage them
git reset HEAD~1               # --mixed is default

# Nuclear: discard last commit AND all changes (irreversible without reflog)
git reset --hard HEAD~1

# Safe undo on shared branch: creates a new "undo" commit
git revert abc1234             # or HEAD for latest
git revert HEAD~3..HEAD        # revert a range

# Discard working tree changes for a file
git restore src/config.js

# Unstage a file (restore from HEAD into index)
git restore --staged src/config.js
Dangerous pattern

git reset --hard followed by git push --force on a shared branch. Anyone who had already pulled now has a diverged history. Use git revert on shared branches — always.

Quick recall

soft = move HEAD only · mixed = HEAD + unstage · hard = HEAD + unstage + discard. revert = safe undo by creating inverse commit. restore = discard working tree changes (not a history operation). Use reset only on local/private branches.

T5

HEAD, detached HEAD, and refs — the pointer system

HEAD is a special ref stored in .git/HEAD. Normally it contains a symbolic ref like ref: refs/heads/main, meaning it points to a branch, which in turn points to a commit. In detached HEAD state, .git/HEAD contains a raw commit SHA directly.

Ref typeLocationDescription
HEAD.git/HEADCurrent position (symbolic or detached)
Branch.git/refs/heads/<name>41-byte text file containing a commit SHA
Remote tracking.git/refs/remotes/origin/<name>Last known state of remote branch
Tag.git/refs/tags/<name>Lightweight: SHA; Annotated: points to tag object
bash — HEAD and refs
# See what HEAD points to
cat .git/HEAD                 # → ref: refs/heads/main (attached)
cat .git/HEAD                 # → abc1234… (detached)

# Enter detached HEAD (useful for inspecting old commits)
git checkout abc1234

# Relative refs
git log HEAD~3..HEAD          # last 3 commits
git show HEAD^                # parent of HEAD (same as HEAD~1)
git show HEAD^^               # grandparent

# In detached HEAD: save work before leaving
git switch -c rescue/experiment   # create branch from current detached HEAD

# A branch is just a file
cat .git/refs/heads/main      # → commit SHA, 41 chars
Key insight

A branch in Git is just a 41-byte text file containing a commit SHA. Creating a branch is O(1) — Git writes one file. This is why branching in Git is "cheap" compared to older VCS systems that copied entire directory trees.

Quick recall

HEAD → branch → commit (normal). HEAD → commit directly (detached). Detached HEAD = no branch moves when you commit; new commits become unreachable after you switch away. Always create a branch to preserve work from detached HEAD state.

T6 Advanced

Interactive rebase — rewriting commit history cleanly

git rebase -i opens an editor with a list of commits and action verbs. You can reorder, combine, edit, or drop commits before pushing. It's the primary tool for cleaning up messy local history before a PR review.

ActionWhat it does
pickKeep the commit as-is
rewordKeep changes, edit the commit message
editStop for amending (add files, split commit)
squashCombine with previous commit, merge messages
fixupCombine with previous commit, discard this message
dropRemove the commit entirely
bash — Interactive rebase workflow
# Rewrite the last 5 commits
git rebase -i HEAD~5

# Squash all feature commits into one before merging
git rebase -i main             # opens editor with all commits ahead of main
# In editor: pick first, squash/fixup the rest

# After rewriting, push safely (you OWN this branch)
git push --force-with-lease    # safer than --force: fails if remote changed

# Abort if things go wrong
git rebase --abort

# Continue after resolving a conflict mid-rebase
git add <resolved-file>
git rebase --continue

# Auto-squash: prefix commits with fixup! or squash! then:
git rebase -i --autosquash HEAD~10
--force-with-lease vs --force

--force overwrites whatever is on the remote unconditionally. --force-with-lease checks that the remote ref matches what you last fetched — if someone else pushed in the meantime, the push is rejected. Always prefer --force-with-lease when force-pushing feature branches.

Quick recall

Interactive rebase = time machine for your own commits. pick/reword/edit/squash/fixup/drop. Only rewrite commits that haven't been shared. Use --force-with-lease not --force. --autosquash with fixup!-prefixed commits automates routine cleanup.

Chapter 3 — Collaboration
T7

Conflict resolution — markers, strategies, and tooling

A conflict occurs when Git can't auto-merge because two branches changed the same region of a file differently. Git marks the file with conflict markers and waits for manual resolution.

text — Conflict markers
<<<<<<< HEAD
const timeout = 5000;           // your version (current branch)
=======
const timeout = 3000;           // incoming version (merging branch)
>>>>>>> feature/faster-timeout
bash — Resolution commands
# Use diff3 style to see the common ancestor (best for context)
git config merge.conflictstyle diff3

# Accept entirely one side (useful for binary files or clear ownership)
git checkout --ours   src/config.js   # keep HEAD version
git checkout --theirs src/config.js   # keep incoming version

# Or with git restore (modern syntax)
git restore --ours   src/config.js
git restore --theirs src/config.js

# Open a visual merge tool
git mergetool                         # uses configured tool (vimdiff, kdiff3, VS Code)

# After resolving, stage and continue
git add src/config.js
git merge --continue                  # or: git rebase --continue

# List all conflicted files
git diff --name-only --diff-filter=U
VS Code as mergetool

git config merge.tool vscode and git config mergetool.vscode.cmd 'code --wait $MERGED'. VS Code shows incoming/current/result in a three-pane view with clickable "Accept" buttons — much faster than editing conflict markers by hand.

Quick recall

diff3 = 3-way view (ours/base/theirs). --ours = HEAD branch, --theirs = the branch being merged in. After resolving: git add then --continue. Never commit with unresolved conflict markers still in the file.

T8

git cherry-pick — applying specific commits across branches

git cherry-pick applies the diff introduced by a specific commit (or range of commits) onto the current branch as a new commit. The new commit gets a different SHA even if content is identical.

bash — cherry-pick patterns
# Pick a single commit
git cherry-pick abc1234

# Pick a range (abc1234 exclusive, def5678 inclusive)
git cherry-pick abc1234..def5678

# -n: stage changes but don't auto-commit (lets you combine or modify)
git cherry-pick -n abc1234
git cherry-pick -n abc1234 def5678    # combine two commits into one

# -x: append the original SHA to the commit message (for traceability)
git cherry-pick -x abc1234
# commit message becomes: "fix: typo (cherry picked from commit abc1234)"

# Typical hotfix workflow: fix on main, cherry-pick to release branch
git checkout main
git cherry-pick <hotfix-sha>

git checkout release/2.1
git cherry-pick <hotfix-sha>         # same fix on release branch

# Resolve conflict during cherry-pick
git cherry-pick --continue
git cherry-pick --abort
When to use cherry-pick

Hotfixes that need to land on multiple maintained branches. When you want one commit from a feature branch without merging the whole branch. Avoid for large ranges — a merge or rebase is cleaner. Cherry-pick doesn't record the branch relationship, so -x is important for audit trails.

Quick recall

cherry-pick = apply one commit's diff to current branch as a new commit. New SHA even if content is same. Use -x to record origin in message. Use -n to stage without committing (for combining). Common pattern: fix on main, cherry-pick to release branches.

T9

git stash — shelving work without committing

git stash saves the working tree and index state onto a stack so you can switch context, then re-apply the saved state later. Stashes are local — they don't push to remote.

bash — Stash commands
# Save with a descriptive message
git stash push -m "WIP: auth middleware refactor"

# Also stash untracked files
git stash push -u -m "includes new files"

# Stash only specific files
git stash push -m "partial" -- src/auth.js src/middleware.js

# Interactive: choose which hunks to stash
git stash push -p

# List all stashes
git stash list
# stash@{0}: On main: WIP: auth middleware refactor
# stash@{1}: On main: includes new files

# Apply and keep on stack
git stash apply stash@{0}

# Apply and remove from stack (most common)
git stash pop

# Create a branch from a stash (best when stash conflicts with current state)
git stash branch feature/auth-wip stash@{0}

# Drop specific stash
git stash drop stash@{1}
git stash clear                      # remove all stashes
Stash branch pattern

If git stash pop conflicts with current branch state, use git stash branch <name>. This checks out the commit where the stash was created, applies the stash there, and creates a branch from it — guaranteed no conflicts.

Quick recall

stash = LIFO stack of WIP saves. push -u includes untracked. push -p interactive. pop = apply + remove. apply = apply + keep. branch = safest recovery from stash. Always use -m message so you remember what each stash is.

T10

fetch vs pull — remote tracking and upstream

git fetch downloads remote objects and updates remote-tracking refs (origin/main) without touching your working tree or local branches. git pull is git fetch + git merge (or rebase if configured).

git fetchgit pull
Updates remote-tracking refsYesYes
Updates local branchNoYes (merge or rebase)
Can create conflictsNoYes
Safe to run anytimeAlwaysOnly on clean working tree
bash — Fetch and pull patterns
# Fetch all remotes, then inspect before integrating
git fetch --all --prune            # --prune removes deleted remote branches
git log HEAD..origin/main          # see what's on remote but not local
git diff HEAD origin/main          # see the diff

# Pull with rebase instead of merge (keeps linear history)
git pull --rebase

# Make rebase the default for all pulls
git config --global pull.rebase true

# Set upstream for a new branch
git push -u origin feature/login   # -u = --set-upstream
git branch -u origin/main          # set upstream for current branch

# See upstream relationships
git branch -vv
Best practice

Prefer git fetch then explicit git merge or git rebase over git pull. You see what's coming before integrating. In CI, always use git fetch — git pull in CI is an antipattern that can cause unexpected merges.

Quick recall

fetch = download only, never touch local branches. pull = fetch + merge (or rebase). pull.rebase true keeps history linear. --prune cleans up stale remote-tracking refs. -u on push sets the upstream relationship.

T11

Tags — lightweight vs annotated, semver, CI/CD triggers

Tags are named pointers to commits. Lightweight tags are just a file with a SHA (like a branch that never moves). Annotated tags are full Git objects with a tagger, timestamp, message, and optional GPG signature — these are the ones to use for releases.

LightweightAnnotated
Stored asRef file (SHA only)Tag object in object store
Has message/taggerNoYes
GPG signableNoYes
Visible in git describeYes (with --tags)Yes (by default)
Use for releasesNoYes
bash — Tagging workflow
# Create annotated tag (always for releases)
git tag -a v2.3.1 -m "Release v2.3.1: fix auth timeout"

# Tag a specific commit (not HEAD)
git tag -a v2.3.0 abc1234 -m "Release v2.3.0"

# Tags are not pushed by default — push explicitly
git push origin v2.3.1             # push single tag
git push origin --tags             # push all tags
git push origin --follow-tags      # push only annotated tags reachable from pushed commits

# List and inspect
git tag -l "v2.*"
git show v2.3.1

# Delete (local and remote)
git tag -d v2.3.1
git push origin :refs/tags/v2.3.1

# git describe: human-readable version from nearest tag
git describe --tags                # → v2.3.1-4-gabcdef1
CI/CD integration

GitHub Actions triggers: on: push: tags: ['v*']. A push of a v* tag triggers the release pipeline. Combined with git push --follow-tags in your release script, this creates a clean "tag = deploy" workflow with zero extra config per release.

Quick recall

Annotated tags for releases (have metadata + signable). Lightweight tags for local bookmarks. Tags must be pushed explicitly (--follow-tags is cleanest). Semver: MAJOR.MINOR.PATCH — bump MAJOR on breaking, MINOR on feature, PATCH on fix.

T12

Git hooks & automation — Husky, lint-staged, commitlint

Git hooks are shell scripts in .git/hooks/ that run at specific lifecycle events. They're not committed by default — Husky solves this by managing hooks through package.json so every developer gets the same hooks after npm install.

HookWhen it runsCommon use
pre-commitBefore commit is createdLint, format, test affected files
commit-msgAfter message is writtenEnforce conventional commit format
pre-pushBefore push to remoteRun full test suite, type checks
prepare-commit-msgBefore editor opensAuto-insert ticket ID from branch name
bash / json — Husky + lint-staged + commitlint setup
# Install
npm install --save-dev husky lint-staged @commitlint/cli @commitlint/config-conventional

# Initialize Husky
npx husky init

# .husky/pre-commit
npx lint-staged

# .husky/commit-msg
npx --no -- commitlint --edit $1

# package.json — lint-staged config
{
  "lint-staged": {
    "*.{ts,js}": ["eslint --fix", "prettier --write"],
    "*.{css,json}": "prettier --write"
  }
}

# commitlint.config.js
module.exports = { extends: ['@commitlint/config-conventional'] };

# Skip hooks for a specific commit (avoid in normal workflow)
git commit --no-verify -m "chore: emergency patch"
lint-staged advantage

lint-staged only lints staged files, not the whole codebase. In a large repo, running ESLint on 3 changed files takes milliseconds vs. 30 seconds for the full codebase. Developers only wait for what they actually changed.

Quick recall

Hooks live in .git/hooks/ (not committed). Husky manages hooks via package.json. pre-commit → lint-staged (fast, only staged files). commit-msg → commitlint (conventional commits). pre-push → full tests. --no-verify bypasses hooks (use only for genuine emergencies).

T13 Advanced

Submodules vs subtrees — embedding external repos

Both let you include one Git repository inside another. They have fundamentally different models for ownership and workflow complexity.

SubmoduleSubtree
How embeddedSeparate repo, pinned to a SHA, referenced in .gitmodulesFiles copied into a subdirectory, history merged
Clone experienceRequires git clone --recurse-submodulesNormal clone — subtree content is just files
Updatesgit submodule update --remotegit subtree pull --prefix=lib/foo ...
Push changes backgit push inside submodule dirgit subtree push --prefix=lib/foo ...
HistorySeparate history, clean separationMixed into parent repo history
Best forShared libraries actively developed by sub-teamThird-party code you rarely update
bash — Submodule and subtree commands
── SUBMODULES ──
git submodule add https://github.com/org/lib.git lib/shared
git clone --recurse-submodules https://github.com/org/main.git
git submodule update --init --recursive   # after cloning without --recurse
git submodule update --remote             # pull latest from tracked branch
git submodule foreach git pull origin main # update all submodules

── SUBTREES ──
# Add (one-time)
git subtree add --prefix=lib/shared https://github.com/org/lib.git main --squash

# Pull updates from upstream
git subtree pull --prefix=lib/shared https://github.com/org/lib.git main --squash

# Push changes back to upstream
git subtree push --prefix=lib/shared https://github.com/org/lib.git main
Submodule pitfall

Submodules are the #1 source of "why doesn't this work after cloning?" issues. Developers forget --recurse-submodules, or submodule SHAs go stale. If your team isn't actively developing the embedded code, subtrees are simpler — there's nothing to forget after a clone.

Quick recall

Submodule = separate repo, pointer in parent, extra clone step required. Subtree = files copied in, normal clone, messier history. Rule of thumb: submodule if you own and actively develop the embedded repo; subtree if you just consume a third-party library.

Branching strategies — Git Flow, GitHub Flow, Trunk-Based

The branching strategy determines how features, releases, and hotfixes flow through the repository. The right choice depends on release cadence, team size, and deployment model.

StrategyBranchesRelease cadenceBest for
Git Flowmain, develop, feature/*, release/*, hotfix/*Scheduled releasesVersioned products, mobile apps, enterprise
GitHub Flowmain + short-lived feature branchesContinuous deploymentSaaS, web apps, small teams
Trunk-BasedSingle main + feature flags, very short branches (<1 day)Multiple deploys per dayHigh-velocity teams, mature CI/CD pipelines
bash — Git Flow pattern
# Git Flow: feature branch workflow
git checkout develop
git checkout -b feature/payment-gateway
# ... work ...
git checkout develop
git merge --no-ff feature/payment-gateway

# Create release branch
git checkout -b release/2.4.0 develop
# ... bug fixes only ...
git checkout main
git merge --no-ff release/2.4.0
git tag -a v2.4.0 -m "Release v2.4.0"

# Hotfix
git checkout -b hotfix/2.4.1 main
# ... fix ...
git checkout main
git merge --no-ff hotfix/2.4.1
git tag -a v2.4.1 -m "Hotfix v2.4.1"
git checkout develop
git merge --no-ff hotfix/2.4.1   # don't lose the fix
bash — GitHub Flow (simpler)
# GitHub Flow: branch from main, PR, merge, deploy
git checkout -b feature/add-dark-mode
# ... work, push, open PR ...
git push -u origin feature/add-dark-mode
# PR reviewed → merge to main → auto-deploy
Trunk-Based key requirement

Trunk-Based Development only works with mature CI/CD (fast tests, green main enforced), feature flags to hide incomplete work, and a team culture of small, frequent commits. Without these, you get broken main regularly.

Quick recall

Git Flow = complex, for versioned releases with long QA cycles. GitHub Flow = simple, for continuous deployment. Trunk-Based = maximum velocity, requires feature flags + strong CI. Most teams using k8s/SaaS should use GitHub Flow or Trunk-Based, not Git Flow.

T15 Advanced

git bisect — binary search for the commit that introduced a bug

git bisect performs a binary search through commit history to find the exact commit that introduced a regression. For 1000 commits, it takes at most 10 steps instead of checking all 1000.

bash — Manual and automated bisect
── MANUAL ──
git bisect start
git bisect bad                      # current commit is broken
git bisect good v2.3.0              # this tag/SHA was working
# Git checks out a commit halfway between good and bad
# Test manually, then:
git bisect good                     # or: git bisect bad
# Repeat until Git reports: "abc1234 is the first bad commit"
git bisect reset                    # return to HEAD

── AUTOMATED (best practice) ──
# Write a test script that exits 0 (good) or non-zero (bad)
cat test-regression.sh
#!/bin/bash
npm test -- --testPathPattern="auth.test" --silent
# exit code 0 = pass = good commit
# exit code 1 = fail = bad commit

git bisect start
git bisect bad HEAD
git bisect good v2.3.0
git bisect run ./test-regression.sh  # fully automated — Git runs the script at each step
git bisect reset
Automated bisect advantage

With git bisect run, Git runs the script at each step automatically and prints the culprit commit without any manual interaction. For a bug reproducible by a test, the whole process takes seconds. The script must be committed before bisecting or use an absolute path.

Quick recall

bisect = binary search, O(log n) steps. start → bad (broken HEAD) → good (last known working) → Git checks out midpoints. Manual: mark good/bad each step. Automated: git bisect run <script> — script exits 0 for good, non-zero for bad. Always git bisect reset when done.

T16 Advanced

Monorepo Git — sparse checkout, partial clone, shallow clone, worktrees

Large monorepos with thousands of files or years of history can make routine Git operations slow. Git provides targeted solutions: sparse checkout (limit working tree), partial clone (limit object download), shallow clone (limit history), and worktrees (multiple checkouts).

FeatureWhat it limitsCommand
Sparse checkoutWorking tree (only checkout specific paths)git sparse-checkout
Partial cloneObject download (skip blobs/trees until needed)git clone --filter=blob:none
Shallow cloneHistory depth (only recent commits)git clone --depth=1
WorktreeN/A (multiple checkouts from one repo)git worktree add
bash — Monorepo Git techniques
── SPARSE CHECKOUT: work on only packages/api ──
git clone --no-checkout https://github.com/org/monorepo.git
cd monorepo
git sparse-checkout init --cone
git sparse-checkout set packages/api shared/utils
git checkout main

── PARTIAL CLONE: skip large blobs at clone time ──
git clone --filter=blob:none https://github.com/org/monorepo.git
# Blobs are downloaded lazily on demand

── SHALLOW CLONE: CI/CD speed ──
git clone --depth=1 https://github.com/org/monorepo.git
# Only the latest commit; no history. Fast but bisect won't work.
git fetch --deepen=100              # extend history when needed

── WORKTREE: work on two branches simultaneously ──
git worktree add ../hotfix-2.1 hotfix/2.1
# ../hotfix-2.1 is a separate directory checked out to hotfix/2.1
# No stashing needed. Each worktree has its own working tree + index.
git worktree list
git worktree remove ../hotfix-2.1
Worktree use case

Worktrees eliminate the git stash / switch / stash pop dance when you need to context-switch mid-feature. Open a second terminal, git worktree add ../fix fix/bug, fix the bug, push, close the terminal. Your feature branch is undisturbed.

Quick recall

sparse-checkout = limit files in working tree (monorepo navigation). partial clone = defer blob download (huge repos). shallow clone = no history (CI speed, but no bisect). worktree = multiple checked-out branches simultaneously from one repo. Combine partial clone + sparse-checkout for fastest monorepo experience.

T17 Advanced

CI/CD & Git — GitHub Actions, branch protection, CODEOWNERS, GitOps

Modern CI/CD is deeply integrated with Git events. Knowing how to configure branch protection rules, CODEOWNERS, and GitOps patterns is expected at senior level.

yaml — GitHub Actions: PR + release pipeline
# .github/workflows/ci.yml
name: CI
on:
  pull_request:
    branches: [main, develop]
  push:
    tags: ['v*']

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0           # full history for git describe/bisect
      - uses: actions/setup-node@v4
        with: { node-version: '20' }

git reflog — recovering lost commits, branches, and rebases

The reflog is Git's safety net. It records every position HEAD and branch refs have pointed to — even after reset --hard, branch deletion, or a bad rebase. Reflog entries expire after 90 days by default.

bash — Reflog recovery scenarios
── SCENARIO 1: Recover from git reset --hard ──
git reset --hard HEAD~3             # oops, lost 3 commits
git reflog                          # find the commit before reset
# HEAD@{1}: commit: feat: add payment flow (sha: abc1234)
git reset --hard abc1234            # restore to that commit

── SCENARIO 2: Recover a deleted branch ──
git branch -D feature/old-work      # accidentally deleted
git reflog | grep "feature/old-work"
# HEAD@{5}: checkout: moving from feature/old-work to main (sha: def5678)
git checkout -b feature/old-work def5678   # recreate from that SHA

── SCENARIO 3: Recover from a bad rebase ──
git rebase main                     # went wrong, commits look mangled
git reflog                          # find the pre-rebase HEAD position
# HEAD@{2}: rebase finished: refs/heads/feature/login (sha: ghi9012)
# HEAD@{3}: checkout: moving from main to feature/login (sha: abc1234)
git reset --hard HEAD@{3}           # restore pre-rebase state

── Reading the reflog ──
git reflog                          # HEAD reflog
git reflog show feature/login       # specific branch reflog
git reflog --date=iso               # show absolute timestamps
Reflog scope

Reflog is local only. It's stored in .git/logs/ and doesn't clone or push. If you've already garbage-collected (git gc) or the entry has expired (90-day default), the commit may be unrecoverable. Always have remote backups of important branches.

Quick recall

reflog = local audit log of every HEAD position. Survives reset --hard, branch deletion, bad rebase. Entries expire in 90 days. Use git reflog to find the SHA, then git reset --hard <sha> or git checkout -b <name> <sha> to recover. Local only — not on remote.

T19 Advanced

Secrets & GPG signing — removing sensitive data, signed commits

Once a secret is pushed to Git history, it's in every clone forever — unless you rewrite history. This requires force-push and re-cloning by all collaborators. Prevention is far better than cure.

bash — Removing secrets from history
── git-filter-repo (recommended — faster, safer than filter-branch) ──
pip install git-filter-repo

# Remove a specific file from all history
git filter-repo --path secrets.env --invert-paths

# Replace a secret value everywhere in history
git filter-repo --replace-text replacements.txt
# replacements.txt:
# AKIAIOSFODNN7EXAMPLE==>REDACTED_AWS_KEY

── BFG Repo Cleaner (Java-based alternative) ──
bfg --delete-files secrets.env
bfg --replace-text passwords.txt

# After either tool, force-push ALL branches and tags
git push --force --all
git push --force --tags

── Rotate the compromised secret immediately ──
# History rewrite does NOT invalidate credentials already seen

── Detect secrets before they're committed ──
# Install gitleaks as a pre-commit hook
gitleaks protect --staged
# or in CI:
gitleaks detect --source=. --report-format=sarif
bash — GPG commit signing
# Generate GPG key
gpg --full-generate-key

# List keys and find your key ID
gpg --list-secret-keys --keyid-format=long
# → sec   4096R/3AA5C34371567BD2

# Configure Git to use it
git config --global user.signingkey 3AA5C34371567BD2
git config --global commit.gpgsign true    # sign all commits
git config --global tag.gpgsign true       # sign all tags

# Add GPG public key to GitHub: Settings → SSH and GPG keys
gpg --armor --export 3AA5C34371567BD2

# Verify a signed commit
git log --show-signature -1
Secret rotation is mandatory

Rewriting history removes the secret from future clones, but anyone who cloned before the rewrite has it. GitHub, npm, AWS all have automated secret scanning that will alert you when a secret is detected. Even after rewriting history, assume the credential is compromised and rotate it immediately.

Quick recall

git-filter-repo > filter-branch (faster, safer). Remove files: --path --invert-paths. Replace values: --replace-text. Always rotate credentials after exposure. gitleaks for pre-commit detection. GPG signing: commit.gpgsign true + key uploaded to GitHub = "Verified" badge on commits.

Git performance — gc, commit-graph, maintenance, large repos

In repos with years of history or thousands of files, routine operations (clone, fetch, log, status) can become slow. Git provides several mechanisms to tune performance without changing workflow.

ProblemCauseFix
Slow git log / git merge-baseNo commit-graph cachegit commit-graph write --reachable --changed-paths
Slow git statusMany loose objects, no index v4git gc, enable core.fsmonitor
Huge .git/objectsLarge binary files committedGit LFS, git filter-repo to purge, git gc --aggressive
Slow cloneFull history + all blobs--depth=1, --filter=blob:none
bash — Performance commands
# Manual GC: pack loose objects, prune dangling
git gc                             # auto threshold: 6700 loose objects
git gc --aggressive                # slower, more compact (use monthly)
git prune                          # remove unreachable objects explicitly

# Commit-graph: accelerates log, merge-base, reachability queries
git commit-graph write --reachable --changed-paths
# Enable auto-update:
git config core.commitGraph true
git config fetch.writeCommitGraph true

# fsmonitor: OS-level file watching for faster git status
git config core.fsmonitor true     # uses fsmonitor daemon (Git 2.37+)

# Scheduled maintenance (combines gc, commit-graph, pack-refs, etc.)
git maintenance start              # registers OS cron job
git maintenance run --task=gc
git maintenance run --task=commit-graph

# Diagnose repo size
git count-objects -vH              # object count and disk usage
git rev-list --all --objects | sort -k 2 | uniq -f 1 -d | \
  git cat-file --batch-check='%(objectsize) %(rest)' | \
  sort -rn | head -20              # find largest objects in history

# Git LFS: store large files outside object store
git lfs track "*.psd" "*.mp4"
git add .gitattributes             # commit the tracking config
Commit-graph impact

The commit-graph is a binary cache of reachability data. On repos with 100k+ commits, operations like git log --graph and git merge-base can go from seconds to milliseconds. Enable fetch.writeCommitGraph true so it updates automatically on every fetch.

Quick recall

gc = pack loose objects + prune. commit-graph = reachability cache (massive speedup for log/merge-base). fsmonitor = OS file watch for faster status. maintenance = scheduled gc + commit-graph + pack-refs. For huge repos: LFS for binaries, partial clone + sparse checkout for checkout speed.
