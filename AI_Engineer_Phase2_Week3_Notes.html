01

The Transformer Architecture

The transformer replaced RNNs in 2017 ("Attention Is All You Need") and nothing has beaten it since. Every major LLM — GPT-4, Claude, Llama, Gemini — is a transformer stack. Understanding it gives you mental models for why LLMs behave the way they do.

The big picture

Two things to memorize about every transformer block: residual connections (output = sublayer(x) + x) prevent vanishing gradients, and LayerNorm stabilizes activations before each sublayer. You learned why these matter in Week 2.

02

Self-Attention

Attention answers one question: for each token, how much should I look at every other token? It does this by computing Query, Key, and Value projections from each token's embedding, then scoring Q against K to produce attention weights over V.

Attention(Q, K, V) = softmax( Q·Kᵀ / √d_k ) · V

Think of it like a search engine: Query = the search term, Key = document metadata, Value = document content. Attention = how well your query matches each key → weighted retrieval of values.

Python · NumPy — Self-attention from scratch
import numpy as np

def softmax(x, axis=-1):
    e = np.exp(x - x.max(axis=axis, keepdims=True))
    return e / e.sum(axis=axis, keepdims=True)

def self_attention(X, W_Q, W_K, W_V):
    """
    X:       (seq_len, d_model) — token embeddings
    W_Q/K/V: (d_model, d_k)    — learned projections
    """
    d_k = W_Q.shape[1]

    Q = X @ W_Q   # "What am I looking for?"    (seq_len, d_k)
    K = X @ W_K   # "What do I contain?"        (seq_len, d_k)
    V = X @ W_V   # "What do I give if chosen?" (seq_len, d_k)

    # Attention scores — how much does token i attend to token j?
    scores  = Q @ K.T / np.sqrt(d_k)   # (seq_len, seq_len)
    weights = softmax(scores)           # rows sum to 1

    # Weighted sum of Values — the actual output
    output  = weights @ V              # (seq_len, d_k)
    return output, weights

# Example: 4-token sequence
np.random.seed(0)
seq_len, d_model, d_k = 4, 8, 4
X   = np.random.randn(seq_len, d_model)
W_Q = np.random.randn(d_model, d_k)
W_K = np.random.randn(d_model, d_k)
W_V = np.random.randn(d_model, d_k)

out, weights = self_attention(X, W_Q, W_K, W_V)
print("Attention weights (each row = one token's attention distribution):")
print(weights.round(3))
# [[0.31, 0.22, 0.29, 0.18],   ← token 0 mostly attends to itself
#  [0.19, 0.41, 0.17, 0.23],   ← token 1 mostly attends to itself
#  ...]

The / √d_k scaling prevents the dot products from growing too large in high dimensions — without it, softmax would push attention toward near-one-hot distributions and gradients would vanish.

Causal masking in GPT-style models

Decoder-only models (GPT, Claude) add a mask that sets future positions to −∞ before softmax, so token 5 cannot attend to tokens 6, 7, 8… This is what makes autoregressive generation possible — the model can only use past context to predict the next token.

03

Multi-Head Attention

Instead of one attention computation, transformers run H parallel attention heads. Each head gets its own W_Q, W_K, W_V — so each learns to attend to different relationship types (syntax, coreference, long-range semantics, etc.) simultaneously.

Python · NumPy — Multi-head attention
def multi_head_attention(X, heads_params, W_O):
    """
    heads_params: list of (W_Q, W_K, W_V) for each head
    W_O:          (H * d_k, d_model) — output projection
    """
    heads = []
    for W_Q, W_K, W_V in heads_params:
        head, _ = self_attention(X, W_Q, W_K, W_V)
        heads.append(head)

    # Concatenate all heads → (seq_len, H * d_k)
    concat = np.concatenate(heads, axis=-1)

    # Final linear projection back to d_model
    return concat @ W_O   # (seq_len, d_model)

# GPT-3 specifics:
# d_model = 12288
# H = 96 heads
# d_k = d_model / H = 128 per head
# Each head independently learns what to attend to
# Head 1 might learn syntax, Head 47 might learn coreference

🔗 Why multiple heads matter

In practice, different heads have been shown to specialize: some track subject-verb agreement, some track pronoun resolution, some track positional patterns. This automatic specialization is not designed in — it emerges from training. It's one of the reasons transformers scale so well: more heads = more types of relationships the model can track simultaneously.

Positional Encoding

Attention is permutation invariant — shuffle the tokens and you get identical attention scores. Positional encodings fix this by adding position information to each token embedding before the first attention layer.

Python · NumPy — Sinusoidal positional encoding (original "Attention Is All You Need")
def positional_encoding(seq_len, d_model):
    """
    PE[pos, 2i]   = sin(pos / 10000^(2i / d_model))
    PE[pos, 2i+1] = cos(pos / 10000^(2i / d_model))
    """
    PE = np.zeros((seq_len, d_model))
    for pos in range(seq_len):
        for i in range(0, d_model, 2):
            angle = pos / (10000 ** (i / d_model))
            PE[pos, i]   = np.sin(angle)
            PE[pos, i+1] = np.cos(angle)
    return PE

# Add to token embeddings before first attention layer:
# X_with_pos = token_embeddings + positional_encoding(seq_len, d_model)

# Modern variants:
# RoPE (Rotary Position Embedding) — used by Llama, Mistral, Claude 3
# ALiBi — linear bias on attention logits (used by BLOOM)
# RoPE is now the dominant choice — it extrapolates better to longer sequences
05

Feed-Forward Network (FFN)

After attention, each token passes independently through a 2-layer FFN. This is where factual knowledge is stored. Attention routes information between tokens; the FFN processes and transforms each token.

Python · NumPy — FFN sublayer
def ffn(x, W1, b1, W2, b2):
    """
    x:     (d_model,) — one token's representation
    d_ffn = 4 * d_model  (typical — quadruples the width)
    GPT-3: d_model=12288 → d_ffn=49152 → d_model=12288
    """
    import torch.nn.functional as F  # GELU in practice
    hidden = gelu(x @ W1 + b1)      # (d_ffn,)
    return hidden @ W2 + b2          # (d_model,)

def gelu(z):
    return 0.5*z*(1+np.tanh(np.sqrt(2/np.pi)*(z+0.044715*z**3)))

# The FFN is where facts live.
# Research has shown you can "edit" a model's factual knowledge
# by targeting specific FFN neurons — this is the basis of
# model editing techniques like ROME and MEMIT.

# Full transformer block (what repeats N times):
def transformer_block(x, attn_params, ffn_params):
    # Sublayer 1: attention + residual + LayerNorm
    x = layer_norm(x + multi_head_attention(x, **attn_params))
    # Sublayer 2: FFN + residual + LayerNorm
    x = layer_norm(x + ffn(x, **ffn_params))
    return x

Embedding layerMaps token IDs to dense vectorsOnce, at input
Self-attentionRoutes information between tokensEvery block, first sublayer
FFNStores + transforms factual knowledgeEvery block, second sublayer
Residual connectionGradient highway — prevents vanishingAround every sublayer
LayerNormNormalizes activations — stabilizes trainingAfter every sublayer
Output headProjects d_model → vocab_size logitsOnce, at output

Tokenization & BPE

Tokenization is the most underestimated topic in LLM engineering. It directly affects API cost, context window usage, model performance on certain tasks, and several surprising failure modes.

What is a token?

A token is not a word. It's a subword unit — typically 3–4 characters on average in English. The model never sees raw text; it sees a sequence of integer IDs.

Python · tiktoken — Token inspection
import tiktoken   # pip install tiktoken

# cl100k_base: used by GPT-4, GPT-3.5, text-embedding-ada-002
enc = tiktoken.get_encoding("cl100k_base")

text   = "The transformer architecture uses self-attention."
tokens = enc.encode(text)
print(tokens)        # [791, 37586, 18112, 5829, 659, 12, 54203, 13]
print(len(tokens))   # 8 tokens for 7 words

# Decode individual tokens — see the exact splits
for tok in tokens:
    print(f"{tok:6d} → '{enc.decode([tok])}'")
# 791    → 'The'
# 37586  → ' transformer'       ← leading space is PART of the token
# 18112  → ' architecture'
# 5829   → ' uses'
# 659    → ' self'
# 12     → '-'                  ← hyphen = own token
# 54203  → 'attention'
# 13     → '.'

The expensive patterns — memorize these

Python · tiktoken — Token cost surprises
# NUMBERS — often 1 digit = 1 token (most expensive ratio)
print(len(enc.encode("1234567890")))   # 6 tokens!
print(len(enc.encode("twelve")))        # 1 token
print(len(enc.encode("$1,234,567.89"))) # 9 tokens
# WHY THIS MATTERS: LLMs struggle with arithmetic partly because
# "127" is tokens ["1","27"] — the model never sees "127" as a unit

# WHITESPACE — leading space = different token!
print(enc.encode("hello"))    # [15339]
print(enc.encode(" hello"))   # [24748]  ← completely different ID
# "Answer:" vs " Answer:" vs "\nAnswer:" — all different tokens
# This is why prompt formatting affects model outputs

# NON-ENGLISH — 3-5x more expensive than English
print(len(enc.encode("Hello")))        # 1 token
print(len(enc.encode("Привет")))       # 3 tokens (Russian)
print(len(enc.encode("こんにちは")))     # 5 tokens (Japanese)
print(len(enc.encode("مرحبا")))        # 4 tokens (Arabic)

# SPECIAL CHARS — each may be its own token
print(len(enc.encode("---")))           # 3 tokens (each dash separate)
print(len(enc.encode("...")))           # 1 token  (ellipsis = single token)

# CODE — generally efficient
print(len(enc.encode("def fibonacci(n):")))  # 5 tokens

How BPE works

Python — BPE algorithm (simplified)
# Byte Pair Encoding — the algorithm behind GPT tokenizers
#
# 1. Start with character-level vocabulary
# 2. Count all adjacent pairs in the training corpus
# 3. Merge the most frequent pair into a new single token
# 4. Repeat ~50,000–100,000 times
#
# Example trace on corpus "low low lower lowest":
#   Start:  l o w   l o w   l o w e r   l o w e s t
#   Step 1: merge ('l','o') → 'lo'
#           lo w   lo w   lo w e r   lo w e s t
#   Step 2: merge ('lo','w') → 'low'
#           low   low   low e r   low e s t
#   Step 3: 'low' is now a single token
#
# Real GPT-4 tokenizer: cl100k_base — 100,277 tokens
# Anthropic uses its own BPE tokenizer (similar approach)

# Count Claude tokens exactly:
import anthropic
client = anthropic.Anthropic()

result = client.beta.messages.count_tokens(
    model="claude-sonnet-4-20250514",
    system="You are a helpful assistant.",
    messages=[{"role": "user", "content": "How does attention work?"}]
)
print(result.input_tokens)   # exact count before sending
The tokenization failure modes you'll hit in production

Counting letters: "How many r's in strawberry?" fails because "strawberry" → ["st","raw","berry"] — the model never sees individual letters. Arithmetic: models struggle because numbers tokenize inconsistently. Non-English RAG: your 200K token context holds far fewer words than you expect for Japanese/Arabic text. Always estimate token counts before designing your context budget.

Context Windows

The context window is the maximum number of tokens the model can see at once — input + output combined. It's your model's working memory. Everything outside it is invisible.

Python — Context window reference and utilities
# Current context windows (2025)
context_windows = {
    "claude-opus-4":    200_000,   # ≈ 150K words = 2 novels
    "claude-sonnet-4":  200_000,
    "claude-haiku-4":   200_000,
    "gpt-4o":           128_000,
    "gemini-1.5-pro": 1_000_000,
}

# Conversion rules of thumb
# 1 token ≈ 4 chars ≈ 0.75 words (English)
# 1 page  ≈ 250 words ≈ 333 tokens
# 1 novel ≈ 90,000 words ≈ 120,000 tokens

def tokens_to_words(t): return t * 0.75
def words_to_tokens(w): return int(w / 0.75)

print(tokens_to_words(200_000))    # ≈ 150,000 words
print(words_to_tokens(80_000))    # ≈ 106,667 tokens (War and Peace)

# Practical budgeting for RAG
def context_budget(model, max_output_tokens=2048):
    total    = context_windows[model]
    system   = 500    # typical system prompt
    reserved = max_output_tokens
    available_for_docs = total - system - reserved
    print(f"Available for retrieved chunks: {available_for_docs:,} tokens")
    print(f"≈ {available_for_docs * 0.75 / 250:.0f} pages of text")

context_budget("claude-sonnet-4")
# Available for retrieved chunks: 197,452 tokens
# ≈ 592 pages of text

The "Lost in the Middle" problem

Research shows LLMs perform significantly worse on information placed in the middle of a long context. Performance is highest at the beginning and end.

RAG implication — critical for your projects

When building RAG systems: put the most relevant retrieved chunks at the start or end of the context, not buried in the middle. Re-rank results so the best chunks come first. Don't dump 50 chunks and expect the model to find the needle at position 40,000.

Temperature, top_p & max_tokens

These three parameters control every API call. Understanding them precisely is the difference between reliable production outputs and flaky, unpredictable ones.

Temperature

Python — Temperature mechanics
import numpy as np

def softmax_with_temp(logits, temperature):
    """Temperature divides logits BEFORE softmax."""
    scaled = logits / temperature
    e = np.exp(scaled - scaled.max())
    return e / e.sum()

# Next-word candidates after "The cat": sat, ran, slept, ate
logits = np.array([2.1, 1.4, 1.1, 0.8])

print(softmax_with_temp(logits, 0.1).round(3))
# [0.997, 0.002, 0.001, 0.000] ← nearly always picks "sat"

print(softmax_with_temp(logits, 1.0).round(3))
# [0.440, 0.219, 0.162, 0.119] ← "sat" likely, others possible

print(softmax_with_temp(logits, 2.0).round(3))
# [0.320, 0.253, 0.231, 0.196] ← nearly uniform, very creative

# temperature=0 is not actually 0 in most APIs — they use greedy decoding
# (always pick the max logit token) or a very small temp like 0.001

top_p — nucleus sampling

Python — Nucleus sampling
def top_p_sampling(probs, p=0.9):
    """
    Sort tokens by probability descending.
    Keep the smallest set whose cumulative prob >= p.
    Sample from that "nucleus".
    """
    sorted_idx   = np.argsort(probs)[::-1]
    sorted_probs = probs[sorted_idx]
    cumulative   = np.cumsum(sorted_probs)

    # First index where cumsum >= p
    cutoff  = np.searchsorted(cumulative, p) + 1
    nucleus = sorted_idx[:cutoff]

    # Renormalize and sample
    nucleus_probs = probs[nucleus] / probs[nucleus].sum()
    return np.random.choice(nucleus, p=nucleus_probs)

# With probs = [0.40, 0.25, 0.20, 0.10, 0.05] and top_p=0.9:
# cumsum = [0.40, 0.65, 0.85, 0.95]
# First time ≥ 0.9: index 3 → sample from top 4 tokens
# The 5th token (0.05 prob) is excluded from consideration

# top_p adapts dynamically: with a flat distribution it includes more tokens,
# with a peaked distribution it includes fewer. This is why many practitioners
# prefer top_p over temperature alone for reliability.

When to use what — production settings

Use casetemperaturetop_pReasoning
Structured data extraction0.01.0Deterministic — same input = same JSON every time
Code generation0.20.95Mostly deterministic, small variation acceptable
Q&A / chat assistant0.70.9Slight creativity, still coherent and on-topic
Brainstorming / ideation1.00.95Full creative range, diverse outputs

Don't set both temperature and top_p high

Temperature and top_p both increase randomness. Using high values for both compounds the effect and makes outputs incoherent. Pick one lever: either turn up temperature or widen top_p — not both. Most production systems tune temperature and leave top_p at 0.9.

System, User & Assistant Roles

Every LLM API structures conversation as a sequence of messages with roles. The system prompt sets identity, rules, and constraints. The user/assistant turns are the conversation. Getting this right is the foundation of all prompt engineering.

Python · Anthropic SDK — Full message structure
import anthropic

client = anthropic.Anthropic()   # reads ANTHROPIC_API_KEY from env

# ── System prompt ─────────────────────────────────────────────
# Sets AI identity, constraints, output format.
# Applied once, persists for the whole conversation.
# NOT part of the messages array in Anthropic's API.
system = """You are a senior Python engineer at a fintech company.
You always:
- Write type-annotated Python 3.11+ code
- Use Pydantic for data validation
- Add brief docstrings
- Raise specific exceptions, never bare except:
Never explain what Python is — assume expert audience."""

response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=512,
    temperature=0.2,
    system=system,
    messages=[
        # ── User turn — human input ───────────────────────────
        {"role": "user",
         "content": "Write a function to validate an IBAN."},

        # ── Assistant prefilling trick (Claude-specific) ──────
        # Putting an incomplete assistant turn forces Claude to
        # CONTINUE from that string — skips preamble entirely.
        {"role": "assistant",
         "content": "def validate_iban(iban: str) -> bool:"},
    ]
)
# Response continues from "def validate_iban..." — no preamble
print(response.content[0].text)

Multi-turn conversation — how memory works

Python — Stateful conversation pattern
def chat(system: str, history: list, user_msg: str) -> str:
    """
    There is no built-in memory — we just send the FULL history
    every time. That's all chatbot "memory" is at the API level.
    """
    history.append({"role": "user", "content": user_msg})

    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=512,
        system=system,
        messages=history        # entire conversation every time
    )

    reply = response.content[0].text
    history.append({"role": "assistant", "content": reply})
    return reply

history = []
system  = "You are a concise ML tutor. Max 2 sentences per answer."

r1 = chat(system, history, "What is gradient descent?")
r2 = chat(system, history, "What is the learning rate?")
r3 = chat(system, history, "What if it is too large?")
# history now has 6 messages — all sent on each call
print(f"Tokens used so far: ~{len(str(history)) // 4}")

# Cost implication: each turn re-sends entire history.
# Long conversations get expensive fast.
# Solution: truncate old turns or summarize history periodically.
The prefilling trick — use it for structured outputs

Starting the assistant turn with "```json\n{" forces Claude to output valid JSON without any preamble. Starting with "def " forces a Python function. This is one of the most effective prompt engineering techniques for reliable structured outputs — more reliable than asking nicely in the user turn.

Streaming Responses

Streaming shows tokens as they're generated instead of waiting for the full response. Essential for any production chatbot — it makes the application feel alive and responsive, and allows early termination if the output is going wrong.

Python · Anthropic SDK — Streaming patterns
import anthropic
client = anthropic.Anthropic()

# ── Basic streaming ────────────────────────────────────────
with client.messages.stream(
    model="claude-sonnet-4-20250514",
    max_tokens=512,
    messages=[{"role": "user", "content": "Explain backpropagation."}]
) as stream:
    for chunk in stream.text_stream:
        print(chunk, end="", flush=True)   # no newline, flush immediately

# ── Streaming with usage stats ─────────────────────────────
with client.messages.stream(
    model="claude-sonnet-4-20250514",
    max_tokens=512,
    messages=[{"role": "user", "content": "Write quicksort in Python."}]
) as stream:
    full_text = ""
    for chunk in stream.text_stream:
        full_text += chunk
        print(chunk, end="", flush=True)

    final = stream.get_final_message()
    cost  = (final.usage.input_tokens  * 0.000003 +
             final.usage.output_tokens * 0.000015)
    print(f"\n\nTokens: {final.usage.input_tokens}in / {final.usage.output_tokens}out")
    print(f"Cost:   ${cost:.5f}")

# ── FastAPI streaming endpoint ─────────────────────────────
from fastapi import FastAPI
from fastapi.responses import StreamingResponse

app = FastAPI()

@app.post("/chat")
async def chat_stream(prompt: str):
    def generate():
        with client.messages.stream(
            model="claude-sonnet-4-20250514",
            max_tokens=1024,
            messages=[{"role": "user", "content": prompt}]
        ) as stream:
            for chunk in stream.text_stream:
                yield f"data: {chunk}\n\n"   # Server-Sent Events
        yield "data: [DONE]\n\n"

    return StreamingResponse(generate(), media_type="text/event-stream")

Token Counting & Cost Estimation

Always estimate cost before running batch jobs. Always count tokens before sending long prompts. Surprise bills are a rite of passage — skip it.

Python — Cost estimation utilities
# Pricing per million tokens (check current docs — prices change)
PRICING = {
    "claude-haiku-4":  {"input": 0.80,  "output": 4.00},
    "claude-sonnet-4": {"input": 3.00,  "output": 15.00},
    "claude-opus-4":   {"input": 15.00, "output": 75.00},
    "gpt-4o":          {"input": 5.00,  "output": 15.00},
    "gpt-4o-mini":     {"input": 0.15,  "output": 0.60},

def estimate_cost(model, input_tokens, output_tokens) -> float:
    p = PRICING[model]
    return (input_tokens * p["input"] + output_tokens * p["output"]) / 1_000_000

# Scale estimations
model = "claude-sonnet-4"
avg_input, avg_output = 500, 300   # tokens per request

for n_requests in [100, 10_000, 1_000_000]:

# 10,000 requests    → $0.60
# 1,000,000 requests → $60.00

        "output_tokens": res.usage.completion_tokens,
        "cost_usd": round(
            (res.usage.prompt_tokens     * 5.0 +
             res.usage.completion_tokens * 15.0) / 1_000_000, 6)
    }

results = []
for p in PROMPTS:
    results.append(call_claude(p))
    results.append(call_gpt4o(p))

# Save CSV
with open("llm_comparison.csv", "w", newline="") as f:
    w = csv.DictWriter(f, fieldnames=results[0].keys())
    w.writeheader(); w.writerows(results)

print(f"Saved {len(results)} results")
print(f"Total cost: ${sum(r['cost_usd'] for r in results):.4f}")

Paste any text, see every token highlighted with a different color and its ID shown. Displays total count, estimated cost, and flags expensive patterns like numbers and non-English text.

import streamlit as st
import tiktoken
import re

enc = tiktoken.get_encoding("cl100k_base")

COLORS = [
    "#2d4a3e", "#3b3468", "#4a2a2a", "#2a3d4a", "#4a3a22",
    "#3a2a4a", "#1e4a38", "#4a2d3a", "#2a4a2d", "#4a3d22",
]

def estimate_cost(n_tokens, model="claude-sonnet-4"):
    return n_tokens * 3.0 / 1_000_000

st.title("Tokenization Visualizer")
st.caption("Paste any text to see how LLMs actually see it")

text = st.text_area("Input text", value="The transformer uses self-attention. Cost: $1,234.", height=120)

if text:
    tokens    = enc.encode(text)
    n_tokens  = len(tokens)
    cost      = estimate_cost(n_tokens)

    col1, col2, col3 = st.columns(3)
    col1.metric("Tokens",   n_tokens)
    col2.metric("Characters", len(text))
    col3.metric("Est. cost", f"${cost:.5f}")

    # Render each token as a colored span
    html_parts = []
    for i, tok_id in enumerate(tokens):
        tok_text = enc.decode([tok_id])
        color    = COLORS[i % len(COLORS)]
        safe     = tok_text.replace("&","&").replace("<","<").replace(">",">")
        html_parts.append(
            f'<span title="{tok_id}" style="background:{color};color:#e6e3dc;'
            f'padding:2px 4px;border-radius:3px;margin:1px;display:inline-block;'
            f'font-family:monospace;font-size:14px">{safe}</span>'
        )
    st.markdown("<div style='line-height:2.2'>" + "".join(html_parts) + "</div>", unsafe_allow_html=True)

    # Flag expensive patterns
    warnings = []
    if re.search(r'\d{4,}', text): warnings.append("⚠ Long numbers detected — expensive to tokenize")
    ratio = n_tokens / max(len(text.split()), 1)
    if ratio > 2.5: warnings.append("⚠ High token-to-word ratio — possible non-English or special chars")
    for w in warnings:
        st.warning(w)

Key mental model to carry forward

An LLM is a function: tokens_in → probability distribution over next token. The transformer architecture is the implementation of that function. Temperature and top_p control how you sample from that distribution. Everything in prompt engineering — few-shot examples, chain-of-thought, structured outputs — is about shaping the input tokens so the output distribution peaks where you want it. You now understand the substrate.
