Chapter 1 — Runtime & Event Loop
T1 Core

Event loop & its six phases — how Node handles async on a single thread

Node.js is single-threaded for JavaScript but handles concurrency via the event loop, powered by libuv — a C++ library that offloads I/O to the OS and provides a thread pool for file system and DNS calls. When async work completes, callbacks are queued; the event loop picks them up phase by phase.

#PhaseWhat runs
1TimerssetTimeout / setInterval callbacks whose delay has elapsed
2Pending I/OI/O error callbacks deferred from the previous cycle
3Idle / PrepareInternal libuv housekeeping — not used in app code
4PollFetches new I/O events; executes I/O callbacks. The loop waits here if nothing else is scheduled
5ChecksetImmediate callbacks
6Close callbackssocket.on('close') and similar close handlers

Microtasks — the critical detail: process.nextTick and Promise callbacks are microtasks. They drain after every phase (and after every individual macrotask) before the loop advances. This gives them higher priority than any macrotask queue.

JavaScript — Priority order
console.log('1 sync');

setTimeout(() => console.log('5 setTimeout'), 0);
setImmediate(() => console.log('4 setImmediate'));
Promise.resolve().then(() => console.log('3 Promise'));
process.nextTick(() => console.log('2 nextTick'));

console.log('1 sync end');
// Output: 1 sync → 1 sync end → 2 nextTick → 3 Promise → 4 setImmediate → 5 setTimeout
Say this in the interview

"Node uses libuv to offload I/O so the single JS thread is never blocked waiting. The event loop has 6 phases — Poll is where it waits for I/O. Microtasks like process.nextTick and Promises are drained after every phase before moving forward, giving them higher priority than timers or setImmediate."

Quick recall

Call stack → nextTick → Promises → setImmediate → setTimeout. The Poll phase is where Node blocks waiting for I/O when idle. libuv = C++ layer that makes the single thread non-blocking.

T2

process.nextTick() vs setImmediate() — when to use each

Featureprocess.nextTick()setImmediate()
QueuenextTick queue (microtask)Check phase (macrotask)
When runsAfter current op, before the loop moves to the next phaseNext iteration — Check phase, after I/O callbacks
PriorityHigher — fires before Promise.then when both queuedLower
Event loop starve riskYes — recursive use blocks all I/O indefinitelySafe for recursive use
JavaScript
// ── Use nextTick: emit event post-constructor so listeners can attach first ──
function MyEmitter() {
  EventEmitter.call(this);
  process.nextTick(() => this.emit('ready')); // constructor finishes first
}

// ── Use setImmediate: chunk CPU-heavy work so I/O still gets turns ──
function heavyTask(items) {
  if (!items.length) return;
  processItem(items.shift());
  setImmediate(() => heavyTask(items)); // yields to I/O between chunks
}

Quick recall

nextTick fires before the loop moves to the next phase — recursive use starves I/O. Use setImmediate for chunking CPU work. Use nextTick to emit events post-constructor so callers can attach listeners before the event fires.

JavaScript — Interop gotchas
// ── __dirname in ESM ──
import { fileURLToPath } from 'url';

const __filename = fileURLToPath(import.meta.url);

T4 Critical

Error handling — operational vs programmer errors, async patterns, production handlers

Two classes to keep separate: Operational errors (expected — 404, validation failures, network timeouts) vs Programmer errors (bugs — undefined.property, wrong type). Handle operational errors gracefully; for programmer errors, log, alert, and restart the process.

JavaScript — Custom error + Express pipeline
// ── 1. Custom AppError class ──
class AppError extends Error {
  constructor(message, statusCode) {
    super(message);
    this.statusCode    = statusCode;
    this.isOperational = true;
    Error.captureStackTrace(this, this.constructor);
  }
}

// ── 2. Async wrapper — catches thrown errors and passes to next(err) ──
const asyncHandler = (fn) => (req, res, next) =>
  Promise.resolve(fn(req, res, next)).catch(next);

router.get('/user/:id', asyncHandler(async (req, res) => {
  const user = await User.findById(req.params.id);
  if (!user) throw new AppError('User not found', 404);
  res.json(user);
}));

// ── 3. Express error middleware — exactly 4 params, registered last ──
app.use((err, req, res, next) => {
  res.status(err.statusCode || 500).json({
    status:  'error',
    message: err.isOperational ? err.message : 'Something went wrong'
  });
});

// ── 4. Global handlers — production critical ──
process.on('unhandledRejection', (reason) => {
  console.error('Unhandled Rejection:', reason);
  server.close(() => process.exit(1)); // graceful shutdown
});

process.on('uncaughtException', (err) => {
  console.error('Uncaught Exception:', err);
  process.exit(1); // MUST exit — process state is unknown
});
Why always process.exit(1) on uncaughtException

After an uncaught exception the process heap may be in a corrupt state — open file handles, half-written data, stale closures. Continuing to serve requests risks silently wrong behaviour. Exit immediately; let PM2 or Kubernetes restart a clean instance within seconds.

Quick recall

uncaughtException = exit immediately. unhandledRejection = graceful shutdown (finish in-flight reqs). Express error handler needs exactly 4 params — Express uses function arity to detect it. asyncHandler wrapper removes the need for try/catch in every route.

T5

EventEmitter — Observer pattern, use cases, and memory leak prevention

EventEmitter implements the Observer / PubSub pattern. It is the backbone of Node.js — streams, HTTP servers, and most core modules extend EventEmitter. Default max listeners per event = 10; exceeding that prints a memory leak warning.

JavaScript — EventEmitter & leak patterns
const EventEmitter = require('events');

class Database extends EventEmitter {
  connect() { setTimeout(() => this.emit('connect', { status: 'ok' }), 1000); }
}
const db = new Database();
db.on('connect', info => console.log('Connected:', info));
db.connect();

// ── Memory leak: new listener added on every request ──
// ✗ 1 000 requests = 1 000 listeners — MaxListeners warning fires
app.get('/data', (req, res) => {
  emitter.on('data', data => res.json(data));
});

// ── Fix: once() auto-removes after first fire ──
app.get('/data', (req, res) => {
  emitter.once('data', data => res.json(data)); // ✓
});

// ── Always clean up in class lifecycle ──
class Connection {
  start() {
    this.handler = data => this.process(data);
    globalEmitter.on('data', this.handler);
  }
  stop() { globalEmitter.off('data', this.handler); } // ✓
}
MethodDescription
.on(event, fn)Add persistent listener
.once(event, fn)One-time listener — auto-removed after first fire
.off(event, fn)Remove a specific listener
.emit(event, ...args)Trigger event synchronously
.removeAllListeners(event)Clear all listeners for an event
.listenerCount(event)Debug — count active listeners
.setMaxListeners(n)Raise the warning threshold (default 10)

Quick recall

Always use once() for one-time events. Always call .off() in destroy/stop methods. Monitor listenerCount() in long-running processes. For cross-service events, prefer RabbitMQ or Kafka over in-process EventEmitter.

Chapter 3 — I/O, Streams & Concurrency
T6 Critical

Streams — four types, pipe(), and backpressure

Streams process data chunk by chunk rather than loading everything into memory. Essential for large files, video streaming, and real-time data pipelines. Backpressure = when a Writable cannot consume data as fast as a Readable produces it. pipe() handles this automatically by pausing the readable when the writable's internal buffer is full and resuming on the drain event.

TypeDescriptionExample
ReadableProduce data in chunksfs.createReadStream()
WritableConsume data in chunksfs.createWriteStream()
DuplexBoth readable and writablenet.Socket
TransformDuplex that modifies data in-flightzlib.createGzip()
JavaScript — pipe() vs manual backpressure
const fs   = require('fs');
const zlib = require('zlib');

// ✗ Without streams — loads entire file into memory
const data = fs.readFileSync('bigfile.txt');

// ✓ pipe() — memory-efficient, backpressure handled automatically
fs.createReadStream('bigfile.txt')
  .pipe(zlib.createGzip())
  .pipe(fs.createWriteStream('bigfile.gz'));

// ── Manual backpressure (what pipe does internally) ──
const readable = fs.createReadStream('huge.file');
const writable = fs.createWriteStream('output.file');

readable.on('data', (chunk) => {
  const canContinue = writable.write(chunk);
  if (!canContinue) {
    readable.pause();                              // slow down source
    writable.once('drain', () => readable.resume()); // resume when ready
  }
});
readable.on('end', () => writable.end());
Production preference

Prefer stream.pipeline() from node:stream/promises over pipe() in production — it properly propagates errors and cleans up all streams in the chain when any one fails. pipe() silently leaves streams open on error.

Quick recall

Streams = chunk-by-chunk, no buffering everything. Backpressure = fast source overwhelms slow sink. pipe() handles it: pause on full buffer, resume on drain. Use pipeline() in production for correct error propagation.

T7

Clustering — how it works and production usage with PM2

Node.js uses one CPU core by default. Clustering forks multiple worker processes — typically one per core — all listening on the same port. The OS distributes connections across workers via round-robin. Each worker is an independent Node.js process with its own V8 heap and event loop; they communicate with the master via IPC.

JavaScript — Cluster module
const cluster = require('cluster');
const http    = require('http');
const os      = require('os');

if (cluster.isMaster) {
  console.log(`Master ${process.pid} running`);
  os.cpus().forEach(() => cluster.fork()); // one worker per CPU core

  cluster.on('exit', (worker) => {
    console.log(`Worker ${worker.process.pid} died — restarting`);
    cluster.fork(); // auto-respawn
  });
} else {
  http.createServer((req, res) => res.end(`Worker: ${process.pid}`)).listen(3000);
}

// ── PM2 abstracts clustering in production ──
// pm2 start app.js -i max      → fork one worker per CPU
// pm2 reload app               → zero-downtime rolling reload
// pm2 logs                     → aggregate logs from all workers
State sharing across workers

Each worker has its own memory. Never store session state, cache, or counters in-process — it will differ across workers. Move shared state to Redis. This includes Socket.io in-memory rooms, rate limiting counters, and distributed locks.

Quick recall

Clustering = one process per core, same port, OS round-robins connections. PM2 abstracts this + handles respawn + zero-downtime reload. Workers share nothing in memory — Redis for shared state.

JavaScript — Worker threads & child process
// ── Worker Threads: CPU-heavy JS ──
const { Worker, isMainThread, parentPort, workerData } = require('worker_threads');

if (isMainThread) {
  const w = new Worker(__filename, { workerData: { nums: [1, 2, 3] } });
  w.on('message', result => console.log(result));
} else {
  parentPort.postMessage(workerData.nums.reduce((a, b) => a + b, 0));
}

// ── Child Process: exec / spawn / fork ──
const { exec, spawn, fork } = require('child_process');

exec('ls -la', (err, stdout) => console.log(stdout));   // buffered output
const ls = spawn('ls', ['-la']);                        // streaming output
ls.stdout.on('data', chunk => console.log(chunk.toString()));

const child = fork('./worker.js');                     // Node child + IPC channel
child.send({ task: 'run' });
child.on('message', result => console.log(result));

Quick recall

These three solve different problems. In a real app you might use all three: PM2 clustering for HTTP, Worker Threads for image resizing, Child Process spawn for calling a Python ML model. The rule: never block the event loop with CPU work.

Chapter 4 — Express, Auth & Security
T9

Middleware in Express — types, order, and the silent next() bug

Middleware is a function with access to req, res, next. Every request passes through a pipeline. Order of app.use() calls is critical.

  • next() — pass control to next middleware in chain
  • next(err) — skip regular middleware, jump to error handler
  • Not calling next() — request hangs silently with no response or error (most common silent bug)
JavaScript — All middleware types & correct stack order
// ── 1. Application-level ──
app.use((req, res, next) => {
  console.log(`${req.method} ${req.url}`);
  next(); // must call next or request hangs forever
});

// ── 2. Router-level ──
const router = express.Router();
router.use((req, res, next) => { next(); }); // scoped to this router only
router.get('/users', handler);
app.use('/api/v1', router);

// ── 3. Error handler — exactly 4 params (Express checks function arity) ──
app.use((err, req, res, next) => {
  res.status(err.statusCode || 500).json({ message: err.message });
});

// ── Production stack: ORDER MATTERS ──
app.use(helmet());                         // 1. Security headers first
app.use(cors(corsOptions));                // 2. CORS
app.use(morgan('combined'));             // 3. Logging
app.use(express.json({ limit: '10kb' })); // 4. Body parsing with size limit
app.use('/api', router);                  // 5. Routes
app.use(errorHandler);                     // 6. Error handler LAST

Quick recall

Security headers first, error handler last. Error middleware needs exactly 4 params — Express uses the arity to detect it. Forgetting next() is the most common silent bug. For async route handlers always wrap in asyncHandler or the error never reaches the error middleware.

Authentication & Authorization — JWT, refresh tokens, RBAC, secure production setup

FeatureSession-basedJWT-based
StateStateful — server stores sessionStateless — client stores token
StorageRedis / DBClient (localStorage or cookie)
ScalabilityNeeds shared store across instancesScales easily — any instance can verify
RevocationEasy — delete session recordHard — need a token blacklist or short TTL
Best forMonoliths / SSR appsAPIs / microservices
JavaScript — JWT + refresh token + RBAC
// ── Login: issue access + refresh token ──
app.post('/login', async (req, res) => {
  const user  = await User.findOne({ email: req.body.email });
  const valid = await bcrypt.compare(req.body.password, user.password);
  if (!valid) return res.status(401).json({ message: 'Invalid credentials' });

  const accessToken  = jwt.sign({ userId: user.id, role: user.role },
    process.env.JWT_SECRET, { expiresIn: '15m' });   // short-lived
  const refreshToken = jwt.sign({ userId: user.id },
    process.env.JWT_REFRESH_SECRET, { expiresIn: '7d' });

  await User.update({ refreshToken }, { where: { id: user.id } }); // store for revocation
  res.cookie('refreshToken', refreshToken,
    { httpOnly: true, secure: true, sameSite: 'strict' }); // inaccessible to JS
  res.json({ accessToken });
});

// ── Auth middleware ──
const authenticate = (req, res, next) => {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ message: 'No token' });
  try {
    req.user = jwt.verify(token, process.env.JWT_SECRET);
    next();
  } catch (err) {
    res.status(401).json({
      message: err.name === 'TokenExpiredError' ? 'Token expired' : 'Invalid token'
    });
  }
};

// ── RBAC authorization ──
const authorize = (...roles) => (req, res, next) => {
  if (!roles.includes(req.user.role))
    return res.status(403).json({ message: 'Forbidden' });
  next();
};
app.delete('/users/:id', authenticate, authorize('admin'), handler);
Security decisions explained

Access tokens are short-lived (15 min) to limit the exposure window if stolen. Refresh tokens in httpOnly cookies prevent XSS theft (JavaScript cannot read them). sameSite: strict eliminates most CSRF. Refresh tokens validated against DB = revocable on logout. bcrypt with saltRounds 12, rate limiting on auth endpoints, account lockout after 5 failures.

Quick recall

Access token = short-lived (15 min), in memory. Refresh token = long-lived (7 d), in httpOnly cookie. httpOnly + sameSite:strict = XSS-safe + CSRF-safe. DB-stored refresh token = revocable.

// ── SQL injection: parameterized queries ──
// ✗ Vulnerable — string interpolation
const q = `SELECT * FROM users WHERE id = ${req.query.id}`;
// ✓ Parameterized
db.query('SELECT * FROM users WHERE id = ?', [req.query.id]);

app.use(xss());                            // 4. XSS input sanitisation

app.use(errorHandler);                     // 8. Error handler last

Quick recall

Helmet for headers. Parameterized queries for SQL injection. xss-clean + mongoSanitize for inputs. sameSite:strict for CSRF. Rate limiting with Redis store for distributed brute force. Body size limit (10kb) prevents payload attacks. Cloudflare at the edge for DDoS.

Chapter 5 — Performance & Production
T12

Performance optimisation — tools, techniques, and production checklist

Start with measurement, not guessing. Use Clinic.js to profile, then tackle the biggest wins first.

JavaScript — Key techniques
// ── 1. Redis caching for hot data ──
app.get('/users', async (req, res) => {
  const cached = await redis.get('all_users');
  if (cached) return res.json(JSON.parse(cached));
  const users = await User.findAll();
  await redis.setex('all_users', 3600, JSON.stringify(users));
  res.json(users);
});

// ── 2. Fix N+1 query problem ──
// ✗ N+1: separate query per user — 1 + N DB round trips
for (const user of users) {
  user.posts = await Post.findAll({ where: { userId: user.id } });
}
// ✓ Eager loading — single JOIN, one DB round trip
const users = await User.findAll({ include: [{ model: Post }] });

// ── 3. Worker Threads for CPU tasks — never block event loop ──
app.get('/resize', (req, res) => {
  const worker = new Worker('./resize.js', { workerData: req.query });
  worker.on('message', result => res.json({ result }));
});

// ── 4. Gzip compression ──
app.use(compression({ level: 6, threshold: 1024 }));
CategoryAction
CachingRedis for hot data, TTL-based invalidation, cache-aside pattern
DatabaseIndexes, pagination, eager loading, connection pooling
CPU tasksWorker Threads — never block the event loop
I/OStreams for large files; never readFileSync in request handlers
NetworkGzip compression, HTTP/2, keep-alive connections
ScalingPM2 cluster mode, horizontal scaling behind a load balancer
ProfilingClinic.js, --prof V8 flag, Datadog APM, EXPLAIN on slow queries

Quick recall

Profile first. Then: Redis cache, fix N+1 queries with eager loading / DataLoader, add DB indexes, move CPU work to Worker Threads, enable gzip. PM2 cluster for multi-core HTTP. Monitor with Datadog or New Relic.

JavaScript — Socket.io production setup
const io = new Server(server, { cors: { origin: 'https://myapp.com' } });

// ── JWT auth middleware for socket connections ──
io.use(async (socket, next) => {
  try {
    socket.user = jwt.verify(socket.handshake.auth.token, process.env.JWT_SECRET);
    next();
  } catch { next(new Error('Auth failed')); }
});

io.on('connection', (socket) => {
  socket.on('room:join',    ({ roomId }) => socket.join(roomId));
  socket.on('message:send', async ({ roomId, message }) => {
    const saved = await Message.create({ roomId, message, userId: socket.user.id });
    io.to(roomId).emit('message:receive', saved);   // broadcast to room
  });
  socket.on('typing:start', ({ roomId }) =>
    socket.to(roomId).emit('typing:show', { user: socket.user.name }));
});

// ── Multi-server scaling: Redis adapter pub/subs events across all instances ──
const { createAdapter } = require('@socket.io/redis-adapter');

T14

Design patterns in Node.js — Singleton, Factory, Repository, Dependency Injection

JavaScript — Core patterns
// ── 1. Singleton — Node module caching gives this for free ──
class Database {
  constructor() {
    if (Database.instance) return Database.instance;
    this.connection = createConnection(process.env.DB_URL);
    Database.instance = this;
  }
}
// Note: requiring the same module twice always returns the cached export

// ── 2. Factory — runtime polymorphism without if/else sprawl ──
class NotificationFactory {
  static create(type) {
    const map = { email: EmailNotification, sms: SMSNotification, push: PushNotification };
    if (!map[type]) throw new Error(`Unknown type: ${type}`);
    return new map[type]();
  }
}
NotificationFactory.create('email').send('Order confirmed!');

// ── 3. Repository — decouple business logic from data layer ──
class MongoUserRepository {
  async findById(id)     { return UserModel.findById(id); }
  async create(data)     { return UserModel.create(data); }
  async update(id, data)  { return UserModel.findByIdAndUpdate(id, data, { new: true }); }
  async delete(id)       { return UserModel.findByIdAndDelete(id); }
}

// Service has zero knowledge of MongoDB — trivial to swap or mock ──
class UserService {
  constructor(userRepository) { this.repo = userRepository; }
  async getUser(id) {
    const user = await this.repo.findById(id);
    if (!user) throw new AppError('Not found', 404);
    return user;
  }
}

// ── 4. Dependency Injection ──
// ✗ Tightly coupled — impossible to unit test without hitting MongoDB

V8 internals & garbage collection — what every Node.js senior must know

V8 compiles JavaScript to machine code in two stages. Understanding this explains why some JS patterns are fast and others silently kill throughput.

StageNameWhat it does
1Ignition (interpreter)Parses JS → bytecode. Fast startup, no optimisation yet.
2TurboFan (JIT compiler)Watches hot functions via profiling feedback. Compiles to optimised machine code for the observed types.
DeoptDeoptimisationIf assumptions change (type changes), TurboFan discards the optimised code and falls back to Ignition.

Hidden classes (Shapes) — V8 assigns a hidden class to every object based on its property layout. Objects with the same properties in the same order share a hidden class, enabling fast property access via a fixed offset.

JavaScript — Hidden classes & deoptimisation
// ✓ Same hidden class — V8 can use fixed offset lookup
const a = { x: 1, y: 2 };
const b = { x: 3, y: 4 };

// ✗ Different hidden class — different property-addition order breaks sharing
const c = { x: 1, y: 2 };
const d = { y: 4, x: 3 }; // y first — new hidden class

// ✗ delete kills the hidden class — fallback to slow dictionary mode
delete c.x;

// ✗ Type change forces deoptimisation
function add(a, b) { return a + b; }
add(1, 2);       // TurboFan optimises for int + int
add('x', 'y');  // type change → deopt → back to Ignition

Garbage collection — two-generation model:

GenerationGC algorithmWhen it runsPause
Young (nursery)Scavenger (Cheney's algorithm — copy-collect)Frequently — new allocations fill it fast~1 ms
Old (tenured)Mark-Sweep-CompactLess often — only objects that survive 2+ young GCs are promoted10–100 ms

Inline caching (IC) states

Monomorphic — one type seen, fully optimised. Polymorphic — 2–4 types, still fast. Megamorphic — 5+ types, V8 gives up optimising, uses generic slow path. Keep hot functions monomorphic: always call them with the same argument shapes.

Prototype pollution — the Node.js-specific attack vector

Prototype pollution is a JavaScript-specific vulnerability where an attacker injects properties into Object.prototype — making those properties appear on all objects in the process. In Node.js this can escalate from a denial-of-service to remote code execution in certain frameworks.

JavaScript — Attack, consequences, and prevention
// ── How it happens: unsafe deep merge of user-controlled input ──
function deepMerge(target, source) {
  for (const key in source) {
    if (typeof source[key] === 'object') {
      target[key] = deepMerge(target[key] || {}, source[key]);
    } else {
      target[key] = source[key]; // ← assigns to __proto__ if key is "__proto__"
    }
  }
}

// Attacker sends: { "__proto__": { "isAdmin": true } }
deepMerge({}, JSON.parse('{"__proto__":{"isAdmin":true}}'));
// Now EVERY plain object in the process has .isAdmin === true
console.log({}.isAdmin); // true — the entire process is compromised

// ── Real consequence in Express ──
// if (req.user.isAdmin) → true for EVERY request, even unauthenticated

// ── Prevention ──

// 1. Validate input with Joi/Zod before merging — reject unknown keys

// 2. Use safe merge (check key is own property, skip __proto__ / constructor)
function safeMerge(target, source) {
  for (const key of Object.keys(source)) { // Object.keys, not for...in
    if (key === '__proto__' || key === 'constructor' || key === 'prototype') continue;
    target[key] = source[key];
  }
}

// 3. Use Object.create(null) for maps — no prototype chain at all
const cache = Object.create(null); // no __proto__, no toString, no hasOwnProperty
cache[userInput] = value;           // safe — there's nothing to pollute

// 4. Object.freeze(Object.prototype) — prevents any additions (breaks some libs)
Object.freeze(Object.prototype);

// 5. Use express-mongo-sanitize to strip $ and . from req.body
// 6. Keep lodash/merge updated — CVE-2019-10744 was a prototype pollution in _.merge
npm supply chain

Prototype pollution vulnerabilities regularly appear in popular npm packages (lodash, hoek, jquery, minimist). Run npm audit in CI. Pin dependency versions in package-lock.json. Use socket.dev or GitHub Dependabot for automated scanning. A transitive dependency 3 levels deep can introduce a pollution vector you never see directly.

Quick recall

Prototype pollution = injecting into Object.prototype via unsafe merge. All objects in the process inherit the injected properties. Prevention: Object.keys() instead of for...in, skip __proto__ / constructor keys, use Object.create(null) for key-value maps, validate all input with Joi/Zod before merging.

HTTP Agent & outbound connection pooling — the silent production killer

Every Node.js app that calls external APIs (databases via HTTP, third-party services, internal microservices) is affected by this. Almost no interview prep material covers it, which is exactly why senior interviewers ask about it.

JavaScript — HTTP Agent internals & configuration
const http  = require('http');
const https = require('https');
const axios = require('axios');

// ── Default globalAgent has maxSockets: Infinity ──
// Under load: 1000 concurrent requests → 1000 TCP connections to same host
// → OS port exhaustion (ephemeral ports run out ~28 000 on Linux defaults)
// → ECONNREFUSED or ETIMEDOUT in production under load

// ✗ Default — new TCP connection per request, no reuse, no limit
axios.get('https://api.stripe.com/charges'); // uses globalAgent

// ✓ Custom agent: connection pooling + keep-alive
const agent = new https.Agent({
  keepAlive:    true,   // reuse TCP connections across requests
  maxSockets:   50,    // max concurrent connections to same host:port
  maxFreeSockets: 10,  // idle connections kept alive in pool
  timeout:      5000,  // socket inactivity timeout
  scheduling:   'lifo' // lifo = reuse recently freed socket (better for keepAlive)
});

// Use with axios: pass as httpsAgent option
const client = axios.create({ httpsAgent: agent });

// Use with native https: pass agent per request
https.get({ hostname: 'api.stripe.com', path: '/charges', agent }, callback);

// ── What keepAlive actually does ──
// Without: TCP handshake (3-way) + TLS handshake (~100ms) per request
// With:    reuse existing TCP+TLS connection — only HTTP overhead per request
// Result:  ~50-150ms latency reduction per outbound call

// ── Diagnosing connection pool exhaustion ──
// Symptom: requests queue up, latency spikes, eventually ETIMEDOUT
// Metric to watch: agent.sockets (active), agent.freeSockets (idle), agent.requests (queued)
setInterval(() => {
  const host = 'api.stripe.com:443';
  console.log({
    active:  Object.keys(agent.sockets[host] || []).length,
    idle:    (agent.freeSockets[host] || []).length,
    queued:  (agent.requests[host]  || []).length,
  });
}, 5000);
Common production incident pattern

App works fine in staging (low concurrency). Goes to production. Under load: all 50 maxSockets are in use, new requests queue. Queue grows if upstream service is slow. Memory grows. Eventually requests timeout. Fix: increase maxSockets for slow upstreams, add circuit breaker (Opossum) so slow service doesn't cascade, and add the queue-length metric to your Datadog dashboard.

Quick recall

Default http.globalAgent = maxSockets: Infinity = unlimited TCP connections = port exhaustion under load. Fix: custom agent with keepAlive: true + bounded maxSockets. keepAlive = skip TCP+TLS handshake on reuse = ~100 ms latency saving per call. AsyncLocalStorage — request context propagation without prop drilling

AsyncLocalStorage (built on AsyncHooks) lets you store context that automatically propagates through all async calls spawned within a given scope — without passing it as a parameter. It is the production solution for request-scoped logging, tracing, and user context in Node.js.

JavaScript — AsyncLocalStorage: request context
const { AsyncLocalStorage } = require('node:async_hooks');
const requestContext = new AsyncLocalStorage();

// ── Middleware: start a new context store for every request ──
app.use((req, res, next) => {
  const store = {
    requestId: req.headers['x-request-id'] || crypto.randomUUID(),
    userId:    req.user?.id,
    startTime: Date.now(),
  };
  requestContext.run(store, next); // all async work from next() shares this store
});

// ── Anywhere deep in the call stack — no parameters needed ──
class UserService {
  async getUser(id) {
    const { requestId, userId } = requestContext.getStore();
    logger.info('Fetching user', { requestId, caller: userId }); // ← always correct
    return db.findById(id);
  }
}

class PaymentRepository {
  async charge(amount) {
    const { requestId } = requestContext.getStore();
    // requestId automatically flows through .then(), await, setTimeout, etc.
    return stripeClient.create(amount, { idempotencyKey: requestId });
  }
}

// ── Works across all async boundaries ──
async function handler(req, res) {
  // requestContext.getStore() returns the same object in:
  // - await calls
  // - .then() callbacks
  // - setTimeout callbacks scheduled from this async tree
  // - EventEmitter handlers triggered from this async tree
  const user = await userService.getUser(req.params.id);
  res.json(user);
}

How it propagates

AsyncLocalStorage uses AsyncHooks under the hood. Every async resource (Promise, setTimeout, stream chunk handler, EventEmitter) tracks which async context created it. When requestContext.run(store, fn) is called, all async resources spawned within fn's async tree — no matter how deep — inherit that store. This is propagated by V8's internal async resource tracking, not by Node.js copying data around.

Interview answer

"How do you get the requestId in a repository function 5 layers deep without passing it as a parameter?" — AsyncLocalStorage. Create one instance per concern (logger, tracing, transaction), set the store in middleware with .run(), read it anywhere with .getStore(). It propagates automatically through all await, Promise.then, and callback boundaries within that async tree.

Quick recall

AsyncLocalStorage = request-scoped global without globals. store.run(value, fn) = set context. store.getStore() = read it anywhere in that async tree. Propagates through await, .then(), setTimeout, EventEmitter automatically. Use cases: request ID, user context, DB transaction, per-request feature flags.
