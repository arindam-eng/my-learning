Chapter 1 — Fundamentals
SD1

What is Scalability? Vertical vs Horizontal scaling — when do you choose each?

Scalability = a system's ability to handle increased load without degrading performance. Vertical = add more resources to one machine. Horizontal = add more machines. Horizontal requires stateless design but has no ceiling.

AspectVertical (Scale Up)Horizontal (Scale Out)
CostExpensive hardwareCheap commodity servers
CeilingHard — max server sizeVirtually unlimited
DowntimeRestart required to upgradeRolling updates, zero downtime
ComplexitySimple — no code changeNeeds stateless design + load balancer
Fault ToleranceSingle point of failureHigh — other servers continue
Use caseDB server, quick fixWeb/API servers, long term
Vertical: [ Server 4GB ] ──upgrade──▶ [ Server 64GB ] hit ceiling eventually Horizontal: ┌─[ Server 1 ] [ Load Balancer ] ───▶ ├─[ Server 2 ] ───▶ [ Redis / Shared DB ] └─[ Server 3 ] Add servers as load grows. App must be stateless.
Key concept

Stateless design is the prerequisite for horizontal scaling — no session data stored on the server. Use Redis for sessions, JWT for auth. Any server must be able to handle any request.

Key interview line

"I start with vertical for speed, but architect for horizontal from day one — stateless services, externalized sessions in Redis, load balancer in front. As traffic grows, horizontal scaling with auto-scaling groups (AWS ASG or K8s HPA) handles load elastically. Adding a new server instance should require zero code changes."

SD2

What is the CAP Theorem? Explain C, A, P — and the difference between ACID and BASE.

In a distributed system you can only guarantee 2 of 3: Consistency, Availability, Partition Tolerance. In practice P is mandatory (networks always fail), so the real choice is CP vs AP.

PropertyACID (Strong Consistency)BASE (Eventual Consistency)
Stands forAtomicity, Consistency, Isolation, DurabilityBasically Available, Soft state, Eventually consistent
ConsistencyImmediate, strictEventual — syncs over time
PerformanceSlower (locking, 2PC)Faster (no global locks)
Use caseBanking, payments, inventorySocial feeds, shopping carts, DNS
ExamplesPostgreSQL, MySQLCassandra, DynamoDB, MongoDB
Consistency ▲ / \ CP / \ CA (CA impossible in distributed sys) / \ Partition ─────── Availability Tolerance [HBase, Zookeeper] [Cassandra, DynamoDB] CP: consistent, AP: always available, may reject reads may return stale data

Key interview line

"For banking I choose CP — I'd rather refuse a request than return a stale balance. For a social feed or shopping cart, AP is fine — briefly stale follower counts don't matter. PACELC extends CAP by also considering latency trade-offs when there's no partition — a more realistic model for production decisions."

SD3

Explain Load Balancing — algorithms, Layer 4 vs Layer 7, and health checks.

A load balancer distributes incoming traffic across backend servers. Choosing the right algorithm and layer depends on request type, session needs, and performance requirements.

AlgorithmHow it worksBest for
Round RobinRequests go to servers in sequence 1,2,3,1,2...Stateless, equal-capacity servers
Weighted Round RobinServers get requests proportional to weightDifferent server capacities
Least ConnectionsSend to server with fewest active connectionsVarying request durations
IP Hashhash(client IP) → same server alwaysSession persistence (sticky sessions)
Least Response TimeLowest latency + fewest connections winsLatency-sensitive apps
Layer 4 vs Layer 7

Layer 4 (Transport): Routes by IP + TCP port. Fast, no content inspection. Use: AWS NLB, game servers.
Layer 7 (Application): Routes by HTTP URL, headers, cookies. Enables path-based routing, SSL termination, A/B testing. Use: AWS ALB, Nginx, Kong.

Key interview line

"I use Layer 7 for web APIs — it enables /api/* → API servers, /static/* → CDN, and A/B routing by header. Least Connections is my default algorithm since request durations vary. Health checks (HTTP GET /health every 5s, 2 consecutive failures = unhealthy) ensure traffic never hits a dead server."

SD4

Explain Caching strategies — Cache-Aside, Write-Through, Write-Behind. What are eviction policies and cache stampede?

Caching reduces DB load and improves latency by storing frequently accessed data in fast memory (Redis). The strategy determines when data enters and leaves the cache.

StrategyHow it worksTrade-off
Cache-AsideApp checks cache → miss → load from DB → populate cacheMost common. Cache only what's requested. Stale on write.
Write-ThroughWrite to cache AND DB synchronouslyAlways consistent. Higher write latency.
Write-BehindWrite to cache, async flush to DB laterFastest writes. Risk of data loss.
Read-ThroughCache fetches from DB on miss automaticallyTransparent to app. Cache library manages it.
Cache-Aside (most common): [ App ] ── 1. GET user:123 ──▶ [ Redis ] ◀── 2. MISS ────────── [ App ] ── 3. SELECT * FROM users ──▶ [ DB ] ◀── 4. row data ──────────── [ App ] ── 5. SET user:123 TTL 300 ──▶ [ Redis ] Next request: cache HIT → skip DB entirely ✓
Eviction PolicyEvictsBest for
LRU — Least Recently UsedOldest accessed itemGeneral purpose (most common)
LFU — Least Frequently UsedLeast accessed countLong-term popular items
TTL — Time To LiveItems past expiryTime-sensitive data
Cache Stampede (Thundering Herd)

A popular key expires → 1000 simultaneous requests miss cache → all hit DB → DB overloads. Fix: Mutex lock (only first request fetches, others wait), TTL jitter (±10% random offset so keys don't expire together), or probabilistic early refresh.

Key interview line

"Cache-Aside is my default — it only caches what's requested and is easy to reason about. I always set TTL + LRU eviction in Redis. For stampede prevention, a mutex with a 50ms timeout means the first request fetches while others wait, then all hit the now-warm cache. Random TTL jitter prevents synchronized expiration of related keys."

Chapter 2 — Scale & Reliability
SD5

Explain Database Sharding and Replication. What are the sharding strategies and challenges of resharding?

Replication = copy same data to multiple servers (read scaling, high availability). Sharding = split data across servers by a shard key (write scaling, storage scaling). Each shard holds a subset of rows.

StrategyHowProsCons
Range-baseduser_id 0–999 → Shard A, 1000–1999 → Shard BSimple, range queries easyHotspot if data is non-uniform
Hash-basedhash(user_id) % N → shardEven distributionRange queries hard; resharding costly
Directory-basedLookup table: user_id → shardFlexible, easy reshardingLookup table = single point of failure
Geo-basedEU users → EU shard, US → US shardLow latency, data residencyUneven if regions differ in size
Shard key: user_id % 3 [ App ] ───▶ [ Shard Router ] ───▶ [ Shard 0 ] user_id: 0,3,6... ───▶ [ Shard 1 ] user_id: 1,4,7... ───▶ [ Shard 2 ] user_id: 2,5,8... Each shard has replicas for HA: [ Shard 0 Master ] ── replicates ──▶ [ Shard 0 Replica ]
  • Hot Shard: one shard gets disproportionate traffic (e.g. celebrity user). Fix: consistent hashing or virtual nodes.
  • Cross-Shard JOINs: require scatter-gather queries — denormalize or use application-level joins.
  • Resharding: adding a shard requires moving data — use consistent hashing to minimize movement.
  • Distributed Transactions: spanning shards requires 2PC — complex and slow, avoid by design.

Key interview line

"I shard only when replication + read replicas + vertical scaling are exhausted — sharding adds massive operational complexity. When sharding, I choose consistent hashing over modulo because adding a shard only remaps 1/N of keys. I design the shard key to avoid hot shards — never use user_id if 1% of users generate 50% of traffic."

SD6

What are Message Queues and Event-Driven Architecture? Compare Kafka vs RabbitMQ.

Message queues decouple producers from consumers — the producer publishes a message without caring who handles it or when. This prevents tight coupling and absorbs traffic spikes naturally.

FeatureApache KafkaRabbitMQ / SQS
ModelLog-based (topics + partitions)Message queue (exchanges + queues)
Message retentionPersisted for configurable time (days/forever)Deleted after acknowledgement
ThroughputMillions/secThousands/sec
Consumer modelPull-based; consumers track their own offsetPush-based; broker tracks delivery
ReplayYes — rewind to any offsetNo — consumed = gone
Best forEvent streaming, audit logs, analytics pipelinesTask queues, RPC, complex routing
Synchronous (coupled) — Email failure breaks Order: [ Order Svc ] ── HTTP ──▶ [ Email Svc ] ── HTTP ──▶ [ Inventory ] Event-Driven (decoupled) — Email down = message queued: [ Order Svc ] ── publish ──▶ [ Kafka ] ── subscribe ──▶ [ Email Svc ] ── subscribe ──▶ [ Inventory Svc ] ── subscribe ──▶ [ Analytics Svc ]
Dead Letter Queue

Failed messages (after N retries) are moved to a DLQ for inspection rather than lost silently. Essential in production. Always configure DLQs and alerts on DLQ depth.

Key interview line

"Kafka for event streaming and replay — user activity tracking, audit logs, analytics pipelines. RabbitMQ/SQS for task queues where once-processed semantics are needed — sending emails, processing payments. The key benefit of event-driven: services are decoupled, you can add new consumers without changing the publisher, and the queue naturally absorbs traffic spikes."

How do you implement Rate Limiting at scale? Explain Token Bucket, Leaky Bucket, Fixed Window, and Sliding Window.

AlgorithmHow it worksProsCons
Fixed Window100 req per 60s. Counter resets at window boundary.Simple, low memoryBoundary burst: 200 req in 2s at window edges
Sliding Window LogTrack timestamp of every request in windowAccurate, no boundary burstHigh memory (stores all timestamps)
Sliding Window CounterWeighted blend of current + previous window countMemory efficient + accurateSlightly approximate
Token BucketTokens added at fixed rate. Requests consume tokens. Burst allowed up to bucket size.Allows bursts, smoothSlightly complex
Leaky BucketRequests fill a queue, processed at constant rate. Excess dropped.Constant output rateNo burst tolerance
Token Bucket: Token Generator: +10 tokens/sec (max 100) ↓ [ Bucket: 75/100 tokens ] ↓ Request arrives ───▶ deduct 1 token ───▶ Allow ✓ Request arrives ───▶ 0 tokens ───▶ Reject 429 ✗ Burst of 75 requests allowed instantly, then 10/sec steady state.
Distributed rate limiting

Single server rate limiting is trivial. Across multiple API servers use a shared counter in Redis + Lua script for atomic check-and-increment. Key format: rate:{user_id}:{window_ts}. Use Redis Sorted Sets for sliding window log.

Key interview line

"For most APIs I use sliding window counter in Redis — accurate, memory-efficient, handles distributed servers. Token bucket is better when burst tolerance is needed — a client accumulates tokens during quiet periods and uses them for a burst. For API gateway-level rate limiting (per user, per plan tier), I configure it at infrastructure level using Kong or AWS API Gateway, keeping rate limit logic out of app code."

SD8

What is a CDN? How does it work, what can you cache, and how do you handle cache invalidation?

A CDN (Content Delivery Network) serves content from edge nodes (Points of Presence) geographically close to the user, reducing latency and offloading origin server traffic.

[ Origin Server ] (your datacenter — single location) ↑ cache miss only [ CDN PoP London ] ───▶ [ EU Users ] 10ms latency [ CDN PoP Mumbai ] ───▶ [ India Users ] 8ms latency [ CDN PoP NYC ] ───▶ [ US Users ] 5ms latency Without CDN: India user → US origin = 200ms+ With CDN: India user → Mumbai PoP = 8ms ✓
Cache decisionExamplesHeader
Cache foreverJS/CSS bundles, images, fonts (content-hashed)Cache-Control: public, max-age=31536000, immutable
Cache with TTLHTML pages, public API responsesCache-Control: public, max-age=300, stale-while-revalidate=60
Never cacheUser-specific data, cart, payment pages, POSTCache-Control: private, no-store
Cache Invalidation Strategies

Versioned URLs: /app.abc123.js — change filename = instant invalidation, forever cache (best).
API Purge: CloudFront/Cloudflare APIs allow instant purge by URL.
Surrogate Keys (Cache Tags): tag responses with entity IDs; purge all responses tagged product:123 on update.

Key interview line

"For a Next.js app: content-hashed filenames for JS/CSS (forever cache), short TTL with stale-while-revalidate for HTML, and Cloudflare cache tags for surgical invalidation when content updates. 'Cache everything, invalidate surgically' gives best performance without stale data risk."

SD9

What is an API Gateway? How does it differ from a reverse proxy and a service mesh?

An API Gateway is the single entry point for all clients — it handles cross-cutting concerns (auth, rate limiting, routing, logging) so individual services don't duplicate this logic.

[ Mobile ] [ Web ] [ 3rd Party ] ↓ ↓ ↓ ────────────┼─────────── ↓ ┌─────────────────────┐ │ API Gateway │ │ Auth Rate-Limit │ │ Route SSL-Terminate │ │ Log Transform │ └─────────────────────┘ ↓ ───────────┬─────────── ↓ ↓ ↓ [ User Svc ] [ Order Svc ] [ Product Svc ]
FeatureReverse Proxy (Nginx)API Gateway (Kong/AWS)Service Mesh (Istio)
ScopeExternal routingExternal + API managementInternal service-to-service
AuthBasicFull (JWT, OAuth, API keys)mTLS between services
Rate LimitingBasicPer-user, per-plan, per-routeBetween services
ObservabilityAccess logsAPI analytics, metricsDistributed tracing, telemetry

Key interview line

"The API Gateway handles cross-cutting concerns so individual services don't duplicate auth, rate limiting, or logging logic. I use AWS API Gateway or Kong for external APIs, and Istio service mesh for internal service-to-service security (mTLS). The BFF pattern (Backend for Frontend) is underused but valuable — your mobile app shouldn't receive the same fat payload as your web app."

SD10

What is Consistent Hashing? How does it solve the resharding problem?

With naive modulo hashing: server = hash(key) % N. Adding 1 server (N → N+1) remaps nearly all keys — massive cache invalidation. Consistent hashing ensures adding/removing a server only remaps K/N keys on average.

Hash ring — keys map to next clockwise server: 0 ───────── 315 │ ring │ 45 │ S1──▶S3 │ 270 │ │ 90 │ S2─── │ ───────── 180 Add S4 between S3 and S1: only keys in that arc remap → minimal disruption. Remove S2: S2's keys go to next clockwise server → only 1/N keys affected. Virtual Nodes: Each server has multiple points on ring → better load distribution, avoids hot spots. → server with 2× capacity gets 2× virtual nodes.

Key interview line

"Consistent hashing is fundamental to distributed caching (Redis Cluster, Memcached) and database sharding. Adding a cache node only invalidates ~1/N of existing cached keys — critical for avoiding cascading DB load during scaling events. Virtual nodes ensure even distribution even when servers have different capacities."

Chapter 3 — Microservices Architecture
SD11

When should you choose Microservices over a Monolith? What are the real costs?

AspectMonolithMicroservices
Team sizeSmall–medium (1 team)Large (multiple teams)
Deploy complexitySimple — one deployComplex — many services, K8s needed
Independent scalingScale all-or-nothingScale per service
Network callsIn-process function callsNetwork calls (latency, failures)
DebuggingSimple stack tracesDistributed tracing required
Data consistencyACID transactionsEventual consistency (SAGA)
Best forStartups, small teams, early productsLarge teams, high scale, org boundaries
Conway's Law

"Organizations design systems that mirror their own communication structure." Microservice boundaries should align with team boundaries. If one team owns everything, a monolith is faster. Extract services when: (1) a module needs independent scaling, (2) team size warrants separation, (3) different deployment cadence is needed.

Key interview line

"Microservices are an organizational solution as much as a technical one — they shine when team topology matches service boundaries (Conway's Law). The hidden costs are real: distributed tracing, service discovery, network partitions, eventual consistency, and K8s complexity. My recommendation: start with a modular monolith, extract services surgically when you hit specific bottlenecks. 'Microservices first' at a startup is premature optimization."

SD13

What is Service Discovery? Explain client-side vs server-side, and Kubernetes DNS.

Services in microservices are dynamic — they scale up/down, crash, move to different IPs. Service Discovery is how a service finds another service without hardcoding IP addresses. A Service Registry maps service name → {healthy instances}.

Client-Side Discovery (Netflix Eureka, Consul): [Service A] ── 1. lookup "order-svc" ──▶ [Registry] ◀── 2. [10.0.1.5:3000, 10.0.1.6:3000] ── [Service A] ── 3. pick + load balance ──▶ [Order Svc Instance] Client is responsible for load balancing logic. Server-Side Discovery (Kubernetes, AWS ELB): [Service A] ── GET order-svc/orders ──▶ [K8s Service / LB] ── route ──▶ [Pod 1 or Pod 2] Infra handles routing. Client calls a stable DNS name.
Kubernetes DNS

Every K8s Service gets a stable DNS name: service-name.namespace.svc.cluster.local. CoreDNS resolves this to the ClusterIP → kube-proxy routes to a healthy pod. No external registry needed — services call each other by name.

Key interview line

"In Kubernetes, service discovery is built-in via DNS — services call each other by name and K8s handles routing to healthy pods. For non-K8s or multi-cloud, Consul provides a full registry with health checking and DNS. I prefer server-side discovery (K8s Services / ELB) because it keeps load-balancing logic out of application code."

SD14

What is the Circuit Breaker pattern? Explain the states and how it prevents cascading failures.

Without a circuit breaker: Service A calls slow Service B → A's threads pile up → A exhausts thread pool → A goes down → services calling A go down → cascading failure. The circuit breaker trips and fast-fails requests, giving Service B time to recover.

CLOSED (normal — requests pass through, failures counted) │ threshold exceeded ↓ OPEN (fast-fail all requests, no wait for timeout) │ timeout expires (e.g. 30s) ↓ HALF-OPEN (probe: let 1 request through) ↓ ↓ success failure ↓ ↓ CLOSED OPEN (reset again) Example thresholds: 50% failure rate in 10s → OPEN. 30s timeout → HALF-OPEN.
PatternPurposeTool
Circuit BreakerStop calling a failing dependency, fast-failResilience4j, Istio
Retry + Exponential BackoffRetry transient failures (1s, 2s, 4s + jitter)Resilience4j, Polly
TimeoutNever wait forever — set aggressive limits (500ms)Any HTTP client
BulkheadIsolate thread pools per dependency — one failure can't exhaust allResilience4j
FallbackReturn cached/default response when circuit is openApplication logic

Key interview line

"Circuit breakers prevent cascading failures — when Service B degrades, the circuit trips and calls to B fail immediately instead of queuing and consuming Service A's thread pool. I combine circuit breaker + retry + timeout + fallback. The fallback is critical — return stale cached data or a degraded response rather than an error when possible. Design for partial failure, not just the happy path."

[ Request ] ──▶ [ Svc A ] ──▶ [ Svc B ] ──▶ [ Svc C ] LOGS — structured events per service "2024-01-15 req_id=abc user=123 action=order_placed latency=45ms" Tools: ELK Stack, Grafana Loki, Datadog Logs METRICS — aggregated numbers over time req/sec, error rate, p95 latency, CPU%, memory, queue depth Tools: Prometheus + Grafana, Datadog Metrics TRACES — full request journey across all services TraceID: abc123 ├── Svc A: 45ms (span: 001) │ └── DB query: 20ms ├── Svc B: 120ms (span: 002) ← bottleneck! └── Svc C: 15ms (span: 003) Tools: Jaeger, Zipkin, Datadog APM, OpenTelemetry

Chapter 4 — Data & Patterns
SD16

What is the SAGA Pattern? How do you handle distributed transactions across microservices without 2PC?

Each service has its own DB. A business transaction spanning multiple services can't use a single ACID transaction. SAGA = a sequence of local transactions, each publishing an event to trigger the next. If one fails, compensating transactions undo previous steps.

Happy path (Choreography): [Order Svc] ── order.created ──▶ [Inventory Svc] ── inventory.reserved ──▶ [Payment Svc] ── payment.charged ──▶ [Notification] Failure path (Payment fails): [Payment Svc] ── payment.failed ──▶ [Inventory Svc] COMPENSATE: release reserved inventory ── inventory.released ──▶ [Order Svc] COMPENSATE: mark order failed
AspectChoreographyOrchestration
ControlDistributed — each service reacts to eventsCentral orchestrator controls flow
CouplingServices know events, not each otherOrchestrator knows all services
VisibilityHard to track overall flowFlow visible in orchestrator dashboard
Best forSimple flows, 2–3 stepsComplex flows, 4+ steps with compensations
ToolsKafka + domain eventsAWS Step Functions, Temporal, Conductor

Key interview line

"SAGA replaces 2PC — synchronous distributed locking that's slow, complex, and a single point of failure. SAGA embraces eventual consistency: each service commits locally and publishes an event. Every step must have a defined compensating transaction. I prefer orchestration for complex flows (4+ steps) because the overall flow is visible in the orchestrator; choreography for simple 2–3 step flows."

SD17

What are CQRS and Event Sourcing? When do you use them and what are the trade-offs?

CQRS = separate the write model (Commands) from the read model (Queries). Event Sourcing = store events that happened, not the current state. Current state = replaying all events.

CQRS: [Client] ── Command: PlaceOrder ──▶ [Write DB] (normalized) │ event sync ↓ ── Query: GetOrderHistory ──▶ [Read DB] (denormalized for display) Event Sourcing — store events, not state: Instead of: UPDATE orders SET status='shipped' Store: { event: OrderPlaced, t1 } { event: OrderShipped, t2 } { event: OrderDelivered, t3 } Current state = replay all events (or use snapshots)
PatternBenefitsCosts
CQRSRead/write models optimized independently. Scale read replicas. Different models per use case.Eventual consistency between write/read. More code.
Event SourcingComplete audit log. Time-travel debugging. Replay events for new projections.Complex queries (must replay). Storage grows. Eventual consistency.

Key interview line

"CQRS is practical when read and write patterns are very different — e.g. a dashboard reading denormalized analytics vs a write-heavy OLTP core. Event Sourcing shines for audit-critical domains (banking, healthcare) — you can replay events to debug exactly what happened. I avoid Event Sourcing unless specifically needed — plain CQRS without event sourcing is often enough and far simpler."

Chapter 5 — System Design Case Studies
SD18

Design a URL Shortener (bit.ly). Walk through requirements, capacity, API, DB schema, and architecture.

Functional: shorten URL → unique short code; redirect short URL → original; custom aliases; expiry. Non-functional: 100:1 read/write ratio; 100M URLs/day writes = ~1160/sec; 10B reads/day = ~115K/sec; redirect latency <100ms.

Write path: [Client] ── POST /shorten ──▶ [API GW] ──▶ [URL Service] ──▶ [ID Gen (Snowflake)] ──▶ [MySQL (write)] ──▶ [Redis (warm cache)] Read path (115K req/sec): [Client] ── GET /abc123 ──▶ [CDN/LB] ──▶ [Redis] HIT → 301/302 immediately MISS → MySQL → populate Redis
DecisionChoiceReason
ID generationTwitter Snowflake → base62 encodeDistributed, unique, sortable, compact 6-char codes
Redirect type302 (temporary)Hits our server for click analytics; 301 would be browser-cached
DB schemashort_code (PK), original_url, user_id, created_at, expires_atPK lookup = O(1) with clustered index
CacheRedis LRU, TTL matches URL expiryHot URLs served in <1ms, skip DB entirely
AnalyticsPublish click events to Kafka asyncNever block the redirect for analytics writes

Key interview line

"I use 302 redirects (not 301) to hit our servers for analytics. The ID generator must be distributed — Snowflake IDs encoded to base62 give unique, sortable, compact codes without a central counter. For analytics (clicks, countries, referrers) I publish click events to Kafka asynchronously — the redirect response must never wait for an analytics write."

SD19

Design a Push Notification System that sends millions of notifications per day across iOS, Android, and Email.

[Services] ── events ──▶ [Kafka: notifications] ↓ ────────────┴──────────── ↓ ↓ [Notif Worker] [Notif Worker] (auto-scale) ↓ ──────────┬──────┴───────────── ↓ ↓ ↓ [ APNs ] [ FCM ] [ SES / SendGrid ] iOS Android Email [Device Token Store: Redis / DynamoDB] user_id → { device_token, platform, preferences, quiet_hours }
  • User Preferences: check notification opt-outs, quiet hours, and frequency caps before sending — not after.
  • Deduplication: use idempotency key (notification_id) to prevent duplicate sends if worker retries.
  • Priority Queues: transactional (OTP, payment receipt) → dedicated high-priority workers that bypass throttling. Marketing → low-priority queue with frequency capping.
  • Provider Rate Limits: APNs and FCM impose per-second limits — implement per-provider throttling in workers.
  • Delivery Tracking: track sent/delivered/opened in a time-series store or analytics platform.

Key interview line

"Decouple notification sending from the triggering service via Kafka — the Order Service publishes an event, the Notification Service consumes it. A slow FCM response never slows down order processing. I separate transactional notifications (OTP, password reset) into a dedicated high-priority queue with dedicated workers that bypass all rate limiting — a delayed OTP is a broken product."

Key interview line

"K8s enables everything microservices promise: independent scaling via HPA, rolling deployments with automatic rollback on failed health checks, service discovery via DNS, and config/secret management. I use Canary for risky changes — route 5% traffic to new version, monitor error rate and p99 latency in Datadog, then progressively roll out. The same container image runs in dev, staging, and prod with different ConfigMaps — immutable infrastructure principle."
